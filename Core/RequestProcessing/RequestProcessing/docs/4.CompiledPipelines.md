# 4. CompiledPipelines

`CompiledPipelines` is a tiny holder for the two `PipelineDelegate`s produced by
[`Build()`](1.PipelineBuilder.md) after the application has configured its command and query pipelines. The
[`RequestDispatcher`](5.RequestDispatcher.md) reads from it at run time to pick the right pipeline per request.

- **Package:** `RaccoonLand.Core.RequestProcessing`
- **Namespace:** `RaccoonLand.Core.RequestProcessing.Pipeline`

## Surface

```csharp
public sealed class CompiledPipelines
{
    public PipelineDelegate? Command { get; set; }
    public PipelineDelegate? Query { get; set; }
}
```

| Member | Description |
| --- | --- |
| `Command` | The compiled command pipeline. Null until built. |
| `Query` | The compiled query pipeline. Null until built. |

## Lifecycle

1. **Startup** — register `CompiledPipelines` as a singleton.
2. **Configuration** — build each pipeline and assign it:

```csharp
compiled.Command = commandBuilder.Use(/* … */).Build();
compiled.Query   = queryBuilder.Use(/* … */).Build();
```

3. **Run time** — the [`RequestDispatcher`](5.RequestDispatcher.md) selects `Command` or `Query` based on the
   request type's pre-registered kind (via [`EndpointInvokerRegistry`](3.EndpointInvokerRegistry.md)) and
   invokes it. If the relevant property is still `null`, the dispatcher throws a clear
   "pipeline has not been built" error.

## Why a mutable holder

Pipelines are composed **once** at startup but consumed on **every** request. Storing the two delegates in a
singleton holder lets the dispatcher (also a singleton) read them with zero per-request allocation, and keeps
the build step (one-time) cleanly separated from dispatch (hot path).

## Host independence

`CompiledPipelines` stores plain `PipelineDelegate`s and has no host dependency. The same holder works whether
the pipelines were configured by an API host or a worker.

See also: [PipelineBuilder](1.PipelineBuilder.md), [RequestDispatcher](5.RequestDispatcher.md).
