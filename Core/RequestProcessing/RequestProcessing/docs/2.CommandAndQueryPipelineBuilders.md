# 2. Command & Query pipeline builders

`CommandPipelineBuilder` and `QueryPipelineBuilder` are the two concrete
[`PipelineBuilder`](1.PipelineBuilder.md) subclasses. They exist so the application can configure **two
independent pipelines** — one for commands (state changes) and one for queries (reads) — each with its own
middleware (for example: validation on commands, caching on queries).

- **Package:** `RaccoonLand.Core.RequestProcessing`
- **Namespace:** `RaccoonLand.Core.RequestProcessing.Pipeline`

## Surface

```csharp
public sealed class CommandPipelineBuilder : PipelineBuilder
{
    public CommandPipelineBuilder(IServiceProvider applicationServices, EndpointInvokerRegistry registry);
}

public sealed class QueryPipelineBuilder : PipelineBuilder
{
    public QueryPipelineBuilder(IServiceProvider applicationServices, EndpointInvokerRegistry registry);
}
```

## The terminal delegate

Both builders pass the same terminal delegate to the base `PipelineBuilder`: look up the invoker registered
for the concrete request type and run it.

```csharp
: base(applicationServices, context => registry.Resolve(context.Request.GetType())(context))
```

The actual work of resolving the endpoint and mapping its `Result` to a `PipelineResponse` lives in the
[`EndpointInvokerRegistry`](3.EndpointInvokerRegistry.md).

## Why two types instead of one

The two pipelines are otherwise identical; they are separate **types** purely so a DI container can register
and resolve each independently, and so a host can configure them separately, e.g.:

```csharp
var commandPipeline = commandBuilder
    .Use(/* logging */)
    .Use(/* validation */)
    .Build();

var queryPipeline = queryBuilder
    .Use(/* logging */)
    .Use(/* caching */)
    .Build();
```

The built delegates are then stored in [`CompiledPipelines`](4.CompiledPipelines.md) for the
[`RequestDispatcher`](5.RequestDispatcher.md) to use at runtime.

## Host independence

Neither builder references ASP.NET Core. The application service provider is passed in as a plain
`IServiceProvider`, so these builders compose pipelines identically in an API or a worker.

See also: [PipelineBuilder](1.PipelineBuilder.md), [EndpointInvokerRegistry](3.EndpointInvokerRegistry.md).
