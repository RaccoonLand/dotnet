# 5. IPipelineBuilder

`IPipelineBuilder` assembles a set of middleware components into a single executable
[`PipelineDelegate`](4.PipelineDelegate.md). It mirrors ASP.NET Core's `IApplicationBuilder`.

- **Package:** `RaccoonLand.Core.RequestProcessing.Abstractions`
- **Namespace:** `RaccoonLand.Core.RequestProcessing.Abstractions.Pipeline`

## Surface

```csharp
public interface IPipelineBuilder
{
    IServiceProvider ApplicationServices { get; }
    IPipelineBuilder Use(Func<PipelineDelegate, PipelineDelegate> middleware);
    PipelineDelegate Build();
}
```

| Member | Description |
| --- | --- |
| `ApplicationServices` | The application (root) service provider — available while composing the pipeline (e.g. to read options). It is **not** the per‑request scope; scoped services are resolved at run time from `PipelineContext.RequestServices`. |
| `Use(middleware)` | Adds a component expressed as `next => context => { … }`. Returns the builder for chaining. |
| `Build()` | Composes all components (plus the terminal endpoint invoker) into one `PipelineDelegate`. |

## Ordering

Components added **first run first** (outermost). Given:

```csharp
builder.Use(A).Use(B).Use(C);
var pipeline = builder.Build();
```

execution nests as `A → B → C → terminal → C → B → A`, i.e. each component wraps the ones added after it. This
is the same mental model as ASP.NET Core middleware ordering.

## The raw `Use` signature

The core `Use(Func<PipelineDelegate, PipelineDelegate>)` is the low‑level primitive: it receives `next` and
returns a new delegate. Most code uses the friendlier overloads in
[`PipelineBuilderExtensions`](6.PipelineBuilderExtensions.md) instead:

```csharp
// Low-level (core interface):
builder.Use(next => async context =>
{
    // before
    await next(context);
    // after
});
```

## Terminal component

`Build()` appends a terminal step that invokes the actual [endpoint](1.Cqrs.md) for `context.Request` and
assigns `context.Response` (mapping the endpoint's [`Result`](2.Result.md) to a [`PipelineResponse`](7.PipelineResponse.md)).
Implementations of `IPipelineBuilder` (provided by a hosting package) own that terminal logic; this
abstraction only defines the contract.

## Design notes

- The builder is a composition tool, used once at startup (per pipeline), not on the request hot path.
- `ApplicationServices` is also used by [`UseMiddleware<T>()`](6.PipelineBuilderExtensions.md) to resolve
  **singleton** middleware instances when `UseMiddleware` is called. Those instances become part of the
  compiled pipeline. Scoped services must still come from
  [`PipelineContext.RequestServices`](3.PipelineContext.md) at run time — never from the middleware constructor.
  See the middleware contract in [PipelineDelegate & IPipelineMiddleware](4.PipelineDelegate.md).

See also: [PipelineBuilderExtensions](6.PipelineBuilderExtensions.md).
