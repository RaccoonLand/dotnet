# 7. PipelineResponse & PipelineMessage

`PipelineResponse` is the default **response envelope** produced by a pipeline. Both successful and failed
results share this single shape, which makes hosts (API, worker, …) able to serialize or act on results
uniformly. It is the type carried by [`PipelineContext.Response`](3.PipelineContext.md): the terminal
[endpoint](1.Cqrs.md) returns a [`Result`](2.Result.md) that is mapped to a `PipelineResponse`, and middleware
may build one to short‑circuit the pipeline.

> Renamed from the original `ApiResponse`/`ApiMessage`. The `Api` prefix was dropped because this abstraction
> is host‑independent — the envelope describes a pipeline result, not specifically an HTTP response.

- **Package:** `RaccoonLand.Core.RequestProcessing.Abstractions`
- **Namespace:** `RaccoonLand.Core.RequestProcessing.Abstractions.Responses`

## Surface

```csharp
public sealed record PipelineResponse
{
    public object? Result { get; init; }
    public IReadOnlyList<PipelineMessage> Errors { get; init; } = [];
    public IReadOnlyList<PipelineMessage> Warnings { get; init; } = [];
    public int? StatusHint { get; init; }
}

public sealed record PipelineMessage(string Code, string Message);
```

| Member | Description |
| --- | --- |
| `Result` | The payload on success; `null` when the request failed or produced no result. |
| `Errors` | Errors that prevented success. **Empty on success.** |
| `Warnings` | Non‑fatal warnings produced while handling the request. Empty when there are none. |
| `StatusHint` | Optional status hint for the host (for example `400`, `401`, `403`). Uses familiar HTTP-style numeric codes as a transport-agnostic convention; not a dependency on ASP.NET Core. When `null`, the host applies its own default mapping. |
| `PipelineMessage(Code, Message)` | A single coded message. `Code` is stable/loggable; `Message` is human‑readable text **supplied by the producer** — this type does not resolve localization itself. |

## Where `Message` comes from

`PipelineMessage` is a passive carrier. Whoever builds it is responsible for the text in `Message`:

| Producer | Typical `Message` content |
| --- | --- |
| **Endpoint / handler** | Literal text, a localization template key, or text already resolved via `IMessageLocalization` in the handler. |
| **[`ExceptionHandlingMiddleware`](../../../Modules/Middlewares/ExceptionHandlingMiddleware/docs/1.ExceptionHandlingMiddleware.md)** | For caught [`DomainException`](../../../Core/Domain/docs/6.DomainException.md)s only: localizes each [`DomainError`](../../../Core/Domain/docs/7.DomainError.md) in `exception.Errors` through `IMessageLocalization` when registered in DI; otherwise uses each error's template key as fallback text. |
| **[`FluentValidationMiddleware`](../../../Modules/Middlewares/FluentValidationMiddleware/docs/1.FluentValidationMiddleware.md)** | Copies each `ValidationFailure.ErrorMessage` as-is — localization is the validator author's responsibility (usually by injecting `IMessageLocalization` into the validator). |

There is **no** localization step inside `RaccoonLand.Core.RequestProcessing.Abstractions`.

> The payload property is named `Result` (not `Response`) so that reading it from the context does not become
> the awkward `context.Response.Response`; it reads as `context.Response.Result`.

## Success vs. failure

| Outcome | `Result` | `Errors` |
| --- | --- | --- |
| Success | the payload (or `null` if void) | empty |
| Domain failure | `null` | one or more messages |

Warnings are orthogonal: a successful response may still carry warnings.

## Examples

```csharp
// Success
var ok = new PipelineResponse { Result = productDto };

// Success with a warning
var okWithWarning = new PipelineResponse
{
    Result = productDto,
    Warnings = [new PipelineMessage("PRODUCT_DEPRECATED", "This product is deprecated.")],
};

// Failure
var failed = new PipelineResponse
{
    Errors = [new PipelineMessage("PRODUCT_NOT_FOUND", "The product was not found.")],
};

// Failure with a status hint (host maps to HTTP 403, for example)
var forbidden = new PipelineResponse
{
    StatusHint = 403,
    Errors = [new PipelineMessage("AUTH_FORBIDDEN", "You do not have access to this endpoint.")],
};

// Assigned to the context by the terminal handler or a middleware:
context.Response = ok;
```

## Why a coded message

Separating a stable `Code` from a localizable `Message` lets callers branch on the code while the text can be
translated per culture. Producers choose whether `Message` is already localized or still a template key;
[`ExceptionHandlingMiddleware`](../../../Modules/Middlewares/ExceptionHandlingMiddleware/docs/1.ExceptionHandlingMiddleware.md)
localizes `DomainException` keys automatically when `IMessageLocalization` is registered. Logs and clients key
off `Code`; humans read `Message`.

## Design notes

- It is a `record`, so it is immutable and value‑comparable; build variants with `with` expressions.
- The message lists default to empty (`[]`), so consumers can iterate without null checks.
- The envelope is intentionally transport‑agnostic; an HTTP host maps it to a status code + body, while a
  worker might log it or publish it — neither is the concern of this abstraction.
- When `StatusHint` is set, middleware or endpoints can suggest a status code (for example `401` for
  unauthenticated, `403` for forbidden). Host adapters should prefer `StatusHint` over their default mapping
  when it is present.
