# 2. Result & Result\<TValue>

`Result` and `Result<TValue>` are what an [endpoint](1.Cqrs.md) returns. They let the developer surface
**errors** and **warnings** from inside a handler â€” explicitly, without throwing â€” instead of being limited to
returning a bare payload. The pipeline terminal maps the result to a
[`PipelineResponse`](7.PipelineResponse.md).

- **Package:** `RaccoonLand.Core.RequestProcessing.Abstractions`
- **Namespace:** `RaccoonLand.Core.RequestProcessing.Abstractions.Results`

## Why this exists

Previously `IEndpoint.ExecuteAsync` returned the raw `TResponse`, so a handler had no way to attach `Errors`
or `Warnings` to the response envelope â€” its only options were "return a value" or "throw". `Result` closes
that gap: a handler can return success (optionally with warnings) or a failure carrying coded errors.

## Surface

```csharp
public sealed record Result
{
    public bool IsSuccess { get; }   // no errors
    public bool IsFailure { get; }
    public IReadOnlyList<PipelineMessage> Errors { get; }
    public IReadOnlyList<PipelineMessage> Warnings { get; }

    public static Result Success();
    public static Result Success(params PipelineMessage[] warnings);
    public static Result Failure(params PipelineMessage[] errors);
    public static Result Failure(IEnumerable<PipelineMessage> errors);
    public Result WithWarning(PipelineMessage warning);
    public Result WithWarnings(IEnumerable<PipelineMessage> warnings);
}

public sealed record Result<TValue>
{
    public bool IsSuccess { get; }
    public bool IsFailure { get; }
    public TValue? Value { get; }
    public IReadOnlyList<PipelineMessage> Errors { get; }
    public IReadOnlyList<PipelineMessage> Warnings { get; }

    public static Result<TValue> Success(TValue value);
    public static Result<TValue> Success(TValue value, params PipelineMessage[] warnings);
    public static Result<TValue> Failure(params PipelineMessage[] errors);
    public static Result<TValue> Failure(IEnumerable<PipelineMessage> errors);
    public Result<TValue> WithWarning(PipelineMessage warning);
    public Result<TValue> WithWarnings(IEnumerable<PipelineMessage> warnings);

    public static implicit operator Result<TValue>(TValue value); // success
}
```

Errors and warnings reuse [`PipelineMessage`](7.PipelineResponse.md) (`Code` + `Message`).

## Key semantics

- **Success is the absence of errors.** `IsSuccess` is derived from `Errors.Count == 0`; there is no separate
  flag to keep in sync.
- **A failure must carry at least one error.** `Failure(...)` throws if given an empty error set.
- **Warnings are orthogonal.** They may accompany a success or a failure.
- **Immutable.** Both are `record`s; `WithWarning(s)` returns a copy.
- **Implicit success.** Because `TValue` converts implicitly to `Result<TValue>`, the happy path stays terse:
  `return dto;`.

## Examples

```csharp
// Happy path â€” implicit conversion from the payload.
public Task<Result<ProductDto>> ExecuteAsync(GetProductByIdQuery request, CancellationToken ct)
    => Task.FromResult<Result<ProductDto>>(dto);

// Success with a warning.
return Result<ProductDto>.Success(dto,
    new PipelineMessage("PRODUCT_DEPRECATED", "This product is deprecated."));

// Or append after the fact.
return Result<ProductDto>.Success(dto)
    .WithWarning(new PipelineMessage("PRODUCT_DEPRECATED", "This product is deprecated."));

// Failure with a coded error (no throw).
return Result<Guid>.Failure(new PipelineMessage("PRODUCT_NOT_FOUND", "The product was not found."));

// Payload-less endpoint.
return Result.Success();
return Result.Failure(new PipelineMessage("ORDER_ALREADY_SHIPPED", "The order was already shipped."));
```

## Mapping to the envelope

A pipeline terminal turns the handler's result into the untyped
[`PipelineResponse`](7.PipelineResponse.md) it stores on `PipelineContext.Response`:

```csharp
context.Response = result.ToPipelineResponse(); // Result or Result<TValue>
```

`ToPipelineResponse` copies `Errors`/`Warnings` and sets `PipelineResponse.Result` to the value on success
(or `null` on failure).

## Relationship to middleware

`Result` is for errors/warnings the **handler itself** decides to return. It complements (does not replace)
the planned crossâ€‘cutting middleware: a validation middleware can shortâ€‘circuit with errors before the
handler runs, and an exceptionâ€‘handling middleware can translate thrown `DomainException`s into a failure
envelope. All three converge on the same `PipelineMessage`/`PipelineResponse` shape.

## Design notes

- Keeping `Result` hostâ€‘independent (it only depends on `PipelineMessage`) means the same handlers and their
  outcomes are reusable in an API, a worker, or tests.
- `Value` is nullable to model failure; on success treat it as present (it mirrors what the handler returned).
