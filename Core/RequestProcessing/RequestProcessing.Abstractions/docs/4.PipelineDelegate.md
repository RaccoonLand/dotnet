# 4. PipelineDelegate & IPipelineMiddleware

These two types define **what a middleware is**. They mirror ASP.NET Core's `RequestDelegate` and
`IMiddleware`, applied to a [`PipelineContext`](3.PipelineContext.md) instead of an `HttpContext`.

- **Package:** `RaccoonLand.Core.RequestProcessing.Abstractions`
- **Namespace:** `RaccoonLand.Core.RequestProcessing.Abstractions.Pipeline`

## Surface

```csharp
public delegate Task PipelineDelegate(PipelineContext context);

public interface IPipelineMiddleware
{
    Task InvokeAsync(PipelineContext context, PipelineDelegate next);
}
```

| Type | Description |
| --- | --- |
| `PipelineDelegate` | A function that processes a context. The composed pipeline is itself one `PipelineDelegate`; each middleware receives the `next` delegate. |
| `IPipelineMiddleware` | A class‑based middleware. Register the type as a **singleton** in DI and add it with `UseMiddleware<T>()`. |

## Middleware lifecycle (RaccoonLand contract)

Unlike ASP.NET Core's scoped `IMiddleware`, RaccoonLand pipeline middleware follows these rules:

1. **Pipeline middlewares are expected to be stateless.**
2. **Request-scoped services must be resolved from `PipelineContext.RequestServices`** inside
   `InvokeAsync` — not from the middleware constructor.
3. **Injecting scoped services into middleware constructors is not supported.**

`UseMiddleware<T>()` resolves the middleware **once** from `IPipelineBuilder.ApplicationServices` when
`UseMiddleware` is **called**. The instance becomes part of the compiled pipeline delegate — it is
infrastructure, not a per-request scoped service.

Constructor injection is limited to **singleton-safe** dependencies (`IOptions<T>`, `ILogger<T>`, …). Endpoints
and other request-scoped services continue to resolve from `context.RequestServices`.

## The `next` convention

A middleware decides whether (and when) to call `next`:

- Call `next(context)` to continue down the pipeline (the common case).
- Skip `next` to **short‑circuit** (for example a cache hit that sets `context.Response` and returns).
- Run code **before** `await next(context)` (inbound) and **after** it (outbound) to wrap the rest of the
  pipeline.

## Class-based middleware example

```csharp
internal sealed class TimingMiddleware(ILogger<TimingMiddleware> logger) : IPipelineMiddleware
{
    public async Task InvokeAsync(PipelineContext context, PipelineDelegate next)
    {
        var start = Stopwatch.GetTimestamp();
        try
        {
            await next(context);
        }
        finally
        {
            logger.LogInformation("{Kind} {Request} took {Elapsed}",
                context.Kind, context.Request.GetType().Name, Stopwatch.GetElapsedTime(start));
        }
    }
}
```

Register it as a **singleton** and add it to the pipeline:

```csharp
services.AddSingleton<TimingMiddleware>();
// ...
builder.UseMiddleware<TimingMiddleware>();
```

## Short-circuiting example

Request-scoped dependencies are resolved inside `InvokeAsync`:

```csharp
internal sealed class CacheMiddleware : IPipelineMiddleware
{
    public async Task InvokeAsync(PipelineContext context, PipelineDelegate next)
    {
        var cache = context.RequestServices.GetRequiredService<IResponseCache>();

        if (context.Kind == RequestKind.Query && cache.TryGet(context.Request, out var cached))
        {
            context.Response = new PipelineResponse { Result = cached };
            return;
        }

        await next(context);
        cache.Set(context.Request, context.Response?.Result);
    }
}
```

## Design notes

- Inline middleware (without a class) is supported through
  [`PipelineBuilderExtensions.Use`](6.PipelineBuilderExtensions.md).
- The envelope set on `context.Response` is a [`PipelineResponse`](7.PipelineResponse.md).
- A future ASP.NET Core-style model where scoped services are injected directly into middleware activation
  would revisit scoped registration; the current design intentionally keeps middleware stateless and compiled.

See also: [PipelineBuilderExtensions](6.PipelineBuilderExtensions.md).
