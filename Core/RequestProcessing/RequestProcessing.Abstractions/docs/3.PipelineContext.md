# 3. PipelineContext

`PipelineContext` is the per‑request object that flows through a request‑processing pipeline. It is the CQRS
counterpart of ASP.NET Core's `HttpContext` — **but with no dependency on ASP.NET Core**, so the same
pipeline and middleware run unchanged in an API, a worker, or a test.

- **Package:** `RaccoonLand.Core.RequestProcessing.Abstractions`
- **Namespace:** `RaccoonLand.Core.RequestProcessing.Abstractions.Pipeline`

## Host independence

The original design wrapped an `HttpContext` and exposed `RequestServices`/`CancellationToken` through it.
That coupled every middleware to ASP.NET Core. This abstraction removes that dependency: the scoped service
provider and the cancellation token are supplied **directly** when the context is created. A host adapter
(for example an ASP.NET Core endpoint, or a worker's message handler) is responsible for building the context
from its own request.

## Surface

```csharp
public enum RequestKind { Command, Query }

public sealed class PipelineContext
{
    public PipelineContext(
        IRequestBase request,
        RequestKind kind,
        IServiceProvider requestServices,
        CancellationToken cancellationToken = default);

    public IRequestBase Request { get; }
    public RequestKind Kind { get; }
    public PipelineResponse? Response { get; set; }
    public IServiceProvider RequestServices { get; }
    public CancellationToken CancellationToken { get; }
    public IDictionary<object, object?> Items { get; }
}
```

| Member | Description |
| --- | --- |
| `Request` | The command or query object being processed (non‑null, implements [`IRequestBase`](1.Cqrs.md)). |
| `Kind` | Whether this request flows through the command or the query pipeline. |
| `Response` | The [`PipelineResponse`](7.PipelineResponse.md) envelope. The terminal handler sets it by wrapping its result; middleware may read it or set it to short‑circuit. `null` until produced. |
| `RequestServices` | The **scoped** service provider for this request — used to resolve endpoints, validators, and any other request-scoped service **inside** middleware or handlers. Not used to resolve pipeline middleware instances (those are singletons, resolved at pipeline build time). |
| `CancellationToken` | Tied to the request lifetime; flows to handlers and I/O. |
| `Items` | A per‑request bag for sharing state across middleware. |

`RequestKind` lets a single pipeline distinguish command vs. query flows (for example to apply a transaction
only to commands).

## Building a context

The pipeline itself never creates the context — the host does. Examples:

```csharp
// In an ASP.NET Core endpoint adapter:
var context = new PipelineContext(
    request,
    RequestKind.Command,
    httpContext.RequestServices,
    httpContext.RequestAborted);

// In a worker / message handler:
using var scope = serviceProvider.CreateScope();
var context = new PipelineContext(
    message,
    RequestKind.Command,
    scope.ServiceProvider,
    stoppingToken);
```

## Design notes

- `Request` and `RequestServices` are validated non‑null in the constructor.
- `Response` is the single mutable hand‑off point between middleware and the terminal handler; everything else
  is immutable for the lifetime of the request except the `Items` bag.
- Keeping the context a plain class (no HTTP types) is what makes the pipeline reusable outside of web hosting.
- Pipeline middleware is **stateless** and resolved at build time; **request-scoped** dependencies are always
  obtained from `RequestServices` inside `InvokeAsync`. See
  [PipelineDelegate & IPipelineMiddleware](4.PipelineDelegate.md).

See also: [PipelineDelegate & IPipelineMiddleware](4.PipelineDelegate.md) and the [PipelineResponse](7.PipelineResponse.md) envelope it carries.
