# 6. PipelineBuilderExtensions

`PipelineBuilderExtensions` provides the ergonomic ways to add middleware to an
[`IPipelineBuilder`](5.IPipelineBuilder.md), mirroring ASP.NET Core's `Use` and `UseMiddleware<T>` helpers.
Custom‑middleware packages build their own `UseXxx()` helpers on top of these.

- **Package:** `RaccoonLand.Core.RequestProcessing.Abstractions`
- **Namespace:** `RaccoonLand.Core.RequestProcessing.Abstractions.Pipeline`

## Surface

```csharp
public static class PipelineBuilderExtensions
{
    public static IPipelineBuilder Use(
        this IPipelineBuilder builder,
        Func<PipelineContext, Func<Task>, Task> middleware);

    public static IPipelineBuilder UseMiddleware<TMiddleware>(this IPipelineBuilder builder)
        where TMiddleware : IPipelineMiddleware;
}
```

| Method | Description |
| --- | --- |
| `Use(context, next)` | Adds an **inline** middleware. Mirrors `app.Use(async (context, next) => { …; await next(); })`. |
| `UseMiddleware<T>()` | Adds a **class‑based** middleware resolved **once at pipeline build time** from `ApplicationServices`. The instance is baked into the compiled delegate. |

## Middleware registration contract

See [PipelineDelegate & IPipelineMiddleware](4.PipelineDelegate.md) for the full rules. In short:

- Register middleware as **singleton** (`AddSingleton` / `TryAddSingleton`).
- Middleware must be **stateless**.
- Resolve request-scoped services from **`PipelineContext.RequestServices`** inside `InvokeAsync`.
- **Do not** inject scoped services into middleware constructors.

## Inline middleware

The `next` here is a parameterless `Func<Task>` that continues the pipeline with the same context:

```csharp
builder.Use(async (context, next) =>
{
    if (context.CancellationToken.IsCancellationRequested)
    {
        return; // short-circuit
    }

    await next(); // continue
});
```

Internally this adapts to the core `Use` primitive:

```csharp
builder.Use(next => context => middleware(context, () => next(context)));
```

## Class-based middleware

`UseMiddleware<T>()` resolves the middleware from `builder.ApplicationServices` when `Build()` runs — not
from the request scope on each invocation:

```csharp
services.AddSingleton<ValidationMiddleware>();
// ...
builder.UseMiddleware<ValidationMiddleware>();
var pipeline = builder.Build(); // middleware instance captured here
```

The type must implement [`IPipelineMiddleware`](4.PipelineDelegate.md) and be registered in DI before the
pipeline is built. Resolution uses `GetRequiredService<TMiddleware>()` — if the type is not registered,
building the pipeline throws a clear DI error.

## Authoring a reusable `UseXxx()` helper

A middleware package typically wraps `UseMiddleware<T>()` so consumers get a one‑liner:

```csharp
public static IPipelineBuilder UseRaccoonLandValidation(this IPipelineBuilder builder)
    => builder.UseMiddleware<ValidationMiddleware>();
```

Registration extension:

```csharp
services.TryAddSingleton<ValidationMiddleware>();
```

## Design notes

- Both helpers ultimately funnel into `IPipelineBuilder.Use`, keeping a single composition primitive.
- Inline middleware is convenient for small, local concerns; class‑based middleware is preferred when the
  logic needs dependencies or unit tests.
- Capturing middleware at build time aligns with RaccoonLand's model: middleware is part of the **compiled
  pipeline**, not part of the request scope.
