# 2. WorkerRequestDispatcher

`WorkerRequestDispatcher` is the worker-side counterpart to HTTP dispatch in
[`RaccoonLandController`](../../AspNetCore/docs/1.RaccoonLandController.md). Each call:

1. Creates a new async DI scope (`IServiceScopeFactory.CreateAsyncScope`).
2. Configures [`WorkerExecutionContext`](3.WorkerExecutionContext.md) from optional metadata.
3. Calls [`IRequestDispatcher.DispatchAsync`](../../RequestProcessing/RequestProcessing.Abstractions/docs/8.IRequestDispatcher.md).
4. Passes the [`PipelineResponse`](../../RequestProcessing/RequestProcessing.Abstractions/docs/7.PipelineResponse.md)
   to [`IPipelineResponseHandler`](4.PipelineResponseHandler.md).

- **Package:** `RaccoonLand.Core.Hosting.Worker`
- **Namespace:** `RaccoonLand.Core.Hosting.Worker`

## Surface

```csharp
public sealed class WorkerRequestDispatcher
{
    public Task<WorkerDispatchResult> DispatchAsync(
        IRequestBase request,
        WorkerExecutionMetadata? metadata = null,
        CancellationToken cancellationToken = default);
}
```

## Why a scope per dispatch

Background workers are long-lived singletons. Handlers, validators, and DbContexts are **scoped**. Opening a
scope per job mirrors the per-request scope of ASP.NET Core and ensures scoped services are not shared across
concurrent or sequential jobs.

## Metadata

Pass [`WorkerExecutionMetadata`](3.WorkerExecutionContext.md) when the job carries user, tenant, or correlation
information (for example from a queue message header). Audit and logging middleware can read it through
`ICurrentExecutionContext` when wired up (see [Registration](5.Registration.md)).

See also: [RaccoonLandBackgroundService](1.RaccoonLandBackgroundService.md).
