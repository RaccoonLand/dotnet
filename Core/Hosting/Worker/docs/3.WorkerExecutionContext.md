# 3. WorkerExecutionContext

`WorkerExecutionContext` implements
[`ICurrentExecutionContext`](../../ExecutionContext/docs/1.ICurrentExecutionContext.md) for a single background
job scope. It is configured from [`WorkerExecutionMetadata`](2.WorkerRequestDispatcher.md) before each dispatch.

- **Package:** `RaccoonLand.Core.Hosting.Worker`
- **Namespace:** `RaccoonLand.Core.Hosting.Worker`

## Surface

```csharp
public sealed record WorkerExecutionMetadata
{
    public string? UserId { get; init; }
    public string? TenantId { get; init; }
    public string? CorrelationId { get; init; }
}

public sealed class WorkerExecutionContext : ICurrentExecutionContext
{
    public bool IsAvailable { get; }
    public string? UserId { get; }
    public string? TenantId { get; }
    public string? CorrelationId { get; }
}
```

`IsAvailable` is `true` when at least one of `UserId`, `TenantId`, or `CorrelationId` was supplied.

## Lifetime

Registered as **scoped** — one instance per job scope created by `WorkerRequestDispatcher`. Not shared across
jobs.

## Combined API + worker hosts

`AddRaccoonLandWorker(wireExecutionContext: true)` registers `ICurrentExecutionContext` →
`WorkerExecutionContext` only when no implementation exists (`TryAdd`). If the API host already registered an
HTTP execution context, pass `wireExecutionContext: false` and provide a composite `ICurrentExecutionContext`
that delegates to the worker context inside worker scopes (application responsibility).

See also: [Registration](5.Registration.md).
