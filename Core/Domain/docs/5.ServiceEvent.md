# 5. ServiceEvent

`ServiceEvent` is the base record for cross-service integration events.

- Namespace: `RaccoonLand.Core.Domain.Events`
- Package: `RaccoonLand.Core.Domain`

## Purpose

Service events are like domain events but their audience is **outside** the service boundary. They are held by an `AggregateRoot<TKey>` and persisted via the Outbox pattern on save, then published to other services / a message broker.

## Surface

| Member | Type | Description |
|--------|------|-------------|
| `EventId` | `Guid` | Unique id (sequential GUID); used for idempotency on the consumer side. |
| `OccurredOnUtc` | `DateTimeOffset` | UTC timestamp of when the event occurred. |
| `EventType` | `string` | The runtime type name; used for serialization/routing. |
| `EventVersion` | `int` | Contract version (default `1`) to support schema evolution over time. |

## Difference from DomainEvent

| | `DomainEvent` | `ServiceEvent` |
|--|---------------|----------------|
| Audience | Inside the service | Other services |
| Versioning | — | `EventVersion` for contract evolution |

## Example

```csharp
public sealed record OrderShippedIntegrationEvent(Guid OrderBusinessKey) : ServiceEvent
{
    // Bump when the published contract changes:
    // public int EventVersion { get; init; } = 2;
}
```

## Notes

- Raise via `AggregateRoot.RaiseServiceEvent(...)`.
- Treat the serialized shape as a public contract; increment `EventVersion` on breaking changes and keep consumers backward compatible where possible.
