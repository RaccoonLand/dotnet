# 4. DomainEvent

`DomainEvent` is the base record for in-domain events raised by an aggregate root.

- Namespace: `RaccoonLand.Core.Domain.Events`
- Package: `RaccoonLand.Core.Domain`

## Purpose

Domain events represent something meaningful that happened **inside** the service boundary. They are held by an `AggregateRoot<TKey>` and persisted via the Outbox pattern when EF Core saves the aggregate, giving at-least-once, transactionally-consistent delivery.

## Surface

| Member | Type | Description |
|--------|------|-------------|
| `EventId` | `Guid` | Unique id (sequential GUID); used for idempotency when consuming the outbox. |
| `OccurredOnUtc` | `DateTimeOffset` | UTC timestamp of when the event occurred. |
| `EventType` | `string` (abstract) | Stable contract name for serialization/routing. Must not come from the CLR type name. |

It is an `abstract record`, so concrete events get value-based equality and `with` expressions for free.

## Example

```csharp
public sealed record OrderSubmitted(Guid OrderBusinessKey) : DomainEvent
{
    public override string EventType => "order.submitted";
}
```

## Notes

- Raise via `AggregateRoot.RaiseDomainEvent(...)`.
- Defaults (`EventId`, `OccurredOnUtc`) are set on construction; override through `init` only when replaying/deserializing.
- Treat `EventType` as a persisted contract: renaming the C# type must not change it. Introduce a new name (and keep consumers compatible) only when the contract itself changes.
- For events crossing the service boundary, use `ServiceEvent` instead (see `5.ServiceEvent.md`).
