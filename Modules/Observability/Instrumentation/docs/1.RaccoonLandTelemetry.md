# 1. RaccoonLandTelemetry

The well-known `ActivitySource` and `Meter` the pipeline instrumentation emits through. The framework only
**produces** telemetry; it never configures exporters. This keeps the package exporter-agnostic — the host
chooses the destination (OTLP, Jaeger, Prometheus, …).

- **Package:** `RaccoonLand.Modules.Observability.Instrumentation`
- **Namespace:** `RaccoonLand.Modules.Observability.Instrumentation.Telemetry`

## Names

| Member | Value | Used for |
| --- | --- | --- |
| `ActivitySourceName` | `RaccoonLand.RequestProcessing` | pipeline spans |
| `MeterName` | `RaccoonLand.RequestProcessing` | pipeline metrics |

## Metrics

| Instrument | Type | Unit | Tags |
| --- | --- | --- | --- |
| `raccoonland.requests.count` | Counter`<long>` | `{request}` | kind, outcome, and optionally name |
| `raccoonland.requests.duration` | Histogram`<double>` | `ms` | kind, outcome, and optionally name |
| `raccoonland.requests.active` | UpDownCounter`<long>` | `{request}` | kind |

`outcome` is one of `success`, `failure` (a domain failure — `PipelineResponse.Errors` is non-empty) or
`exception` (an unhandled exception bubbled out of the pipeline). A missing/`null` pipeline response is treated
as `success` (there are no domain errors to report).

The optional `raccoonland.request.name` tag on count/duration is controlled by
`InstrumentationOptions.RequestNameInMetrics` (`None` / `Name` / `FullName`). The default is `FullName` so
requests that share a short type name across namespaces do not collide. Use `Name` or `None` when you need
to reduce metric cardinality.

## Span tags

| Tag | Source |
| --- | --- |
| `raccoonland.request.kind` | `Command` / `Query` |
| `raccoonland.request.name` | `request.GetType().FullName` (always; independent of metrics tagging) |
| `raccoonland.request.outcome` | `success` / `failure` / `exception` |
| `enduser.id` | `ICurrentExecutionContext.UserId` |
| `raccoonland.tenant.id` | `ICurrentExecutionContext.TenantId` |
| `raccoonland.correlation_id` | `ICurrentExecutionContext.CorrelationId` |

### Span status vs outcome

Only an **unhandled exception** sets the span status to `Error`. The status description is a fixed generic
message (not `exception.Message`) to avoid leaking sensitive detail into status fields; the exception is still
attached via `Activity.AddException` for exporters that consume exception events.

**Domain failures** (non-empty `PipelineResponse.Errors`) keep an unset span status. They are distinguished
only by the tag `raccoonland.request.outcome = failure`. That is intentional: a business/validation failure is
not a span-level system error.

## Host wiring (the host's job)

```csharp
builder.Services.AddOpenTelemetry()
    .WithTracing(t => t
        .AddSource(RaccoonLandTelemetry.ActivitySourceName)
        .AddOtlpExporter())
    .WithMetrics(m => m
        .AddMeter(RaccoonLandTelemetry.MeterName)
        .AddPrometheusExporter());
```
