# 2. PipelineInstrumentationMiddleware

A single `IPipelineMiddleware` that instruments every command/query with the three observability pillars at
once, so the pipeline is only walked once:

- **Tracing** — starts a span `"{Kind} {RequestFullName}"` on the shared `ActivitySource`, tags it, records the
  outcome, and on an exception sets the span status to `Error` (generic description; no `exception.Message`)
  and attaches the exception via `AddException`. Domain failures do **not** set span status to `Error`.
- **Metrics** — increments an in-flight up/down counter for the duration, then records a request count and a
  duration histogram tagged by kind, outcome, and (optionally) request name per `RequestNameInMetrics`.
- **Logging** — when `EnableLogging` is true, opens an `ILogger` scope (`CorrelationId`, `UserId`, `TenantId`,
  `TraceId`) so **every** log written during the request carries them, writes a start (`Debug`) entry, and on
  completion either a finish entry (`Information` for success, `Warning` for domain failure) or, on an
  unhandled exception, a single `Error` entry in the catch path (no second finish log). When
  `EnableLogging` is false, this middleware emits **no** logs at all, including on exceptions.

- **Package:** `RaccoonLand.Modules.Observability.Instrumentation`
- **Namespace:** `RaccoonLand.Modules.Observability.Instrumentation.Diagnostics`

## Why this is the framework's job (not the host's)

The CQRS pipeline (`PipelineContext`, `IPipelineMiddleware`) and the notion of a correlation id are framework
abstractions, so instrumenting them belongs to the framework. The host remains responsible for the transport
concerns: installing the OpenTelemetry SDK and exporters, configuring sampling and resource attributes
(`service.name`, `service.version`), wiring Serilog sinks, and mapping endpoints. HTTP-server logging/metrics
and runtime/EF Core/HttpClient instrumentation are likewise the host's job.

## Behavior

- Each pillar can be disabled independently via `InstrumentationOptions` (`EnableTracing`, `EnableMetrics`,
  `EnableLogging`). Disabling logging turns off start, finish, and exception logs from this middleware.
- `RequestNameInMetrics` (`None` / `Name` / `FullName`) controls whether count/duration metrics include
  `raccoonland.request.name`. Spans always use the request type's `FullName`. Default is `FullName` (opt into
  `Name` or `None` to lower cardinality). Unsupported values are rejected at host startup by options
  validation; the middleware itself never throws for a bad tag value and falls back to `FullName` if needed.
- Options are resolved at the **start** of each request. A single snapshot is held for that request. Valid
  reloads apply to subsequent requests. If a reload fails options validation, the middleware keeps the last
  known-good snapshot — an `OptionsValidationException` from `IOptionsMonitor` must not fail the business
  request. (Initial invalid config still fails host startup via `ValidateOnStart`.)
- The middleware is a singleton. `ILogger` is injected (singleton-safe); the request-scoped
  `ICurrentExecutionContext` is resolved per request from `PipelineContext.RequestServices`.
- Exceptions are re-thrown after being recorded, so the rest of the pipeline (for example the exception-handling
  middleware) behaves exactly as before.

## Placement

Add it **first** (outermost) so it measures the whole pipeline, including downstream middleware:

```csharp
services.AddRaccoonLandRequestProcessing(
    configureCommandPipeline: pipeline =>
    {
        pipeline.UseMiddleware<PipelineInstrumentationMiddleware>();
        // … other middleware (validation, authorization, …)
    },
    configureQueryPipeline: pipeline =>
    {
        pipeline.UseMiddleware<PipelineInstrumentationMiddleware>();
        // …
    });
```
