# 3. IFileUrlSigner

`IFileUrlSigner` issues direct HTTP read/write URLs for providers that support storage-native signing (S3-compatible object stores).

- **Package:** `RaccoonLand.Modules.FileStorage.Abstractions`
- **Namespace:** `RaccoonLand.Modules.FileStorage.Abstractions`
- **Source:** `src/Contracts/IFileUrlSigner.cs`

## Surface

```csharp
public interface IFileUrlSigner
{
    Task<SignedUrlResult> GetReadUrlAsync(
        SignedReadUrlRequest request,
        CancellationToken cancellationToken = default);

    Task<SignedUrlResult> GetWriteUrlAsync(
        SignedWriteUrlRequest request,
        CancellationToken cancellationToken = default);
}
```

## When to use

Use signed URLs when clients should transfer bytes directly to or from object storage without streaming through your API host. Typical cases:

- Browser or mobile direct upload/download
- CDN-friendly read paths
- Large files where proxying through the app is expensive

Do **not** use signed URLs as a substitute for authorization. Issue URLs only after your application has verified the caller may access the file.

## Request types

**Read**

```csharp
public sealed class SignedReadUrlRequest
{
    public required string Key { get; init; }
    public TimeSpan? Expiry { get; init; }
    public string? ContentType { get; init; }  // ignored for S3 GET signing (see below)
}
```

**Write**

```csharp
public sealed class SignedWriteUrlRequest
{
    public string? Key { get; init; }          // generated when null
    public required string ContentType { get; init; }
    public long? MaxSizeBytes { get; init; }
    public TimeSpan? Expiry { get; init; }
    public IReadOnlySet<string>? AllowedContentTypes { get; init; }
    public long? MaxUploadBytes { get; init; }
}
```

Expiry is clamped by [FileStorageOptions](5.FileStorageOptions.md) (`DefaultSignedUrlExpiry`, `MaxSignedUrlExpiry`).

**S3 signing notes:** `GetWriteUrlAsync` includes `Content-Type` in the signed headers so the client must send the same type on PUT. `GetReadUrlAsync` does **not** sign `Content-Type` — browsers and download clients typically omit that request header, and locking it would cause `SignatureDoesNotMatch` (403). The returned read `FileRef` therefore carries only `Key` (not request `ContentType`). Persist MIME in application state or resolve via `GetMetadataAsync` when needed.

Build write requests with [FileStoragePutHelper.CreateSignedWriteRequest](11.FilePutConstraints.md) for per-endpoint content type and size guardrails.

## Result

```csharp
public sealed record SignedUrlResult(Uri Url, DateTimeOffset ExpiresAtUtc, FileRef File);
```

Return the `FileRef` from write URLs to the client only when appropriate — usually you persist `File.Key` server-side after the client confirms upload completion.

## Registration

Only the S3 provider registers this interface. Resolve it optionally:

```csharp
var signer = serviceProvider.GetService<IFileUrlSigner>();
if (signer is null)
{
    // FileSystem / SqlServer — proxy through IFileStorage instead
}
```

FileSystem and SqlServer providers do not implement URL signing.
