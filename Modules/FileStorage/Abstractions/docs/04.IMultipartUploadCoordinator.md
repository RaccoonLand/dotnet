# 4. IMultipartUploadCoordinator

`IMultipartUploadCoordinator` orchestrates multipart uploads for large direct-to-storage transfers.

- **Package:** `RaccoonLand.Modules.FileStorage.Abstractions`
- **Namespace:** `RaccoonLand.Modules.FileStorage.Abstractions`
- **Source:** `src/Contracts/IMultipartUploadCoordinator.cs`

## Surface

```csharp
public interface IMultipartUploadCoordinator
{
    Task<MultipartUploadSession> InitiateAsync(
        InitiateMultipartUploadRequest request,
        CancellationToken cancellationToken = default);

    Task<UploadPartResult> UploadPartAsync(
        UploadPartRequest request,
        CancellationToken cancellationToken = default);

    Task<FileRef> CompleteAsync(
        CompleteMultipartUploadRequest request,
        CancellationToken cancellationToken = default);

    Task AbortAsync(
        AbortMultipartUploadRequest request,
        CancellationToken cancellationToken = default);
}
```

## Lifecycle

```text
InitiateAsync  →  UploadPartAsync (repeat per part)  →  CompleteAsync
                  └─ AbortAsync on failure/cancel
```

1. **Initiate** — reserve a key and receive an `UploadId`.
2. **Upload parts** — stream each part; collect `UploadPartResult` (part number + ETag).
3. **Complete** — commit parts into a final [FileRef](2.FileRef.md). The returned `FileRef` may omit `ContentType` and `Length` (S3 Complete does not return them). Persist MIME from Initiate in the application, or call `IFileStorage.GetMetadataAsync` afterward.
4. **Abort** — discard an incomplete session (call on client cancel or unrecoverable errors).

## Request and result types

| Type | Role |
|------|------|
| `InitiateMultipartUploadRequest` | Optional key, required content type, optional metadata |
| `MultipartUploadSession` | `(Key, UploadId)` session handle |
| `UploadPartRequest` | Part number (1-based), content stream, optional length |
| `UploadPartResult` | `(PartNumber, ETag)` returned per part |
| `CompleteMultipartUploadRequest` | Key, upload id, ordered parts list (no content type — MIME is set at initiate) |
| `AbortMultipartUploadRequest` | Key and upload id |

## When to use

Prefer multipart uploads when:

- Files exceed single-request upload limits
- Clients upload in parallel chunks
- You need resumable upload flows

For small files, `IFileStorage.PutAsync` is simpler.

## Registration

Only the S3 provider registers `IMultipartUploadCoordinator` alongside `IFileStorage` and `IFileUrlSigner`. Other providers throw [FileStorageNotSupportedException](8.Exceptions.md) if multipart is attempted.

## Application responsibilities

- Track in-progress uploads in application storage if clients may retry across requests.
- Enforce authorization before initiating or completing uploads.
- Abort orphaned sessions (background job or expiry policy).
- Do not expect `PutMode.CreateOnly` semantics here — multipart has no create-only mode. Prefer generating a new key per upload, or best-effort `ExistsAsync` before initiate (still subject to races).
- After `CompleteAsync`, `FileRef.ContentType` and `FileRef.Length` are typically unset. Persist content type from `InitiateAsync` in application state, and/or call `IFileStorage.GetMetadataAsync` when full metadata is required.
