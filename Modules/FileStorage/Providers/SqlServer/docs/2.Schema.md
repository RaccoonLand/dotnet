# 2. Schema

When `EnsureSchema` is `true`, the provider creates the schema (if missing) and table:

```sql
CREATE SCHEMA [app]; -- only when SchemaName is not dbo and schema does not exist

CREATE TABLE [app].[FileBlobs] (
    [Key] NVARCHAR(128) NOT NULL PRIMARY KEY,
    [Content] VARBINARY(MAX) NOT NULL,
    [ContentType] NVARCHAR(128) NULL,
    [Length] BIGINT NOT NULL,
    [ChecksumSha256] NVARCHAR(64) NULL,
    [CreatedAtUtc] DATETIMEOFFSET NOT NULL,
    [Metadata] NVARCHAR(MAX) NULL
);
```

Default `SchemaName` is `dbo`. Set both `SchemaName` and `TableName` in configuration to control the qualified object name.

For production deployments, prefer managing this schema through your application migration tooling and set `EnsureSchema` to `false`.

When `EnsureSchema` is `true` but the connection lacks DDL permission, the provider raises `FileStorageConfigurationException` with an actionable message instead of surfacing a raw `SqlException`. That configuration failure is cached so later requests do not repeat the DDL attempt.

## Put modes

| Mode | SQL behaviour |
|------|---------------|
| `CreateOnly` | `INSERT`; fails when the key exists |
| `Overwrite` | `UPDATE` only; fails with `FileNotFoundStorageException` when the key is missing |
| `Upsert` | `UPDATE`; `INSERT` when no row was updated |

`CreatedAtUtc` is set only on insert. Overwrite and upsert updates preserve the original creation timestamp, matching the FileSystem provider. Put operations return the stored `CreatedAtUtc` via an `OUTPUT` clause.

Concurrent upserts on the same new key may hit a primary-key conflict; the provider retries as an update.

Put operations that emit multiple result sets (for example upsert falling through to insert) iterate all OUTPUT result sets before concluding that no timestamp was returned.
