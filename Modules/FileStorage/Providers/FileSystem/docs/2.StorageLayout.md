# 2. Storage layout

How the FileSystem provider maps opaque storage keys to files on disk.

- **Package:** `RaccoonLand.Modules.FileStorage.FileSystem`
- **Source:** `src/Internal/FileSystemPathLayout.cs`

## Directory structure

Objects are sharded by the first four characters of the storage key to avoid large flat directories:

```text
{RootPath}/
└── file/                          ← shard = first 4 chars of key
    ├── file_0195abcd...           ← object content (binary)
    └── file_0195abcd....meta.json ← sidecar metadata (JSON)
```

Example for key `file_0195abcd0123456789abcdef01234567`:

| File | Path |
|------|------|
| Content | `{RootPath}/file_/file_0195abcd0123456789abcdef01234567` |
| Metadata | `{RootPath}/file_/file_0195abcd0123456789abcdef01234567.meta.json` |

Keys shorter than four characters use the full key as the shard folder name.

## Metadata sidecar

Each upload writes a `.meta.json` file alongside the content file:

```json
{
  "key": "file_0195abcd0123456789abcdef01234567",
  "contentType": "image/png",
  "length": 2048,
  "checksumSha256": "a1b2c3...",
  "createdAtUtc": "2026-07-15T12:00:00+00:00",
  "userMetadata": {
    "source": "avatar-upload"
  }
}
```

`FileRef.Version` and `FileRef.ChecksumSha256` are populated from the SHA-256 computed during write.

## Write behaviour

1. Stream content to a temporary file (`{object}.tmp-{guid}`) in the target shard directory.
2. Validate size against [FileStorageOptions](../../Abstractions/docs/05.FileStorageOptions.md).
3. Write metadata to a temporary sidecar (`{object}.meta.json.tmp-{guid}`).
4. Commit the object file:
   - `CreateOnly` — atomic `File.Move(..., overwrite: false)`; fails with `FileAlreadyExistsStorageException` if the key already exists (no TOCTOU pre-check).
   - `Overwrite` / `Upsert` — replace the existing object file.
5. Commit the metadata sidecar with atomic `File.Move(..., overwrite: true)`.
6. Consistency rules after the object is committed:
   - `CreateOnly` — if metadata commit fails, the object file is rolled back (deleted) and the exception propagates.
   - `Overwrite` / `Upsert` — metadata sidecar is **best-effort**. If metadata commit fails, `PutAsync` still succeeds and returns the in-memory `FileRef`; reads fall back to object-only metadata until a later successful write refreshes the sidecar.

`CreatedAtUtc` is stored in the sidecar. On overwrite/upsert, the previous sidecar timestamp is preserved when present.

Partial writes never replace the live object — failed uploads discard temp files in a `finally` block inside the write helpers.

## Read behaviour

- `OpenReadAsync` opens the content file as a read-only `FileStream`. Dispose the `OpenReadResult` to release the handle.
- `GetMetadataAsync` reads the sidecar when present. When the sidecar is missing or unreadable (corrupt/truncated JSON, I/O errors, or deleted between existence check and open), returns length only — **no** `CreatedAtUtc` fallback from filesystem timestamps. Corrupt or transiently unreadable sidecars are treated the same as missing sidecars so a valid object file remains readable.

## Delete behaviour

`DeleteAsync` removes the metadata sidecar first, then the content file. If the object delete fails, metadata is already gone but the object remains — `ExistsAsync` still returns true and a retry can complete deletion. When `IgnoreNotFound` is true (default), missing objects are a no-op.

## Operational notes

- Ensure the process identity has read/write access to `RootPath`.
- Back up `{RootPath}` as a unit — keys in your application database map to files under this tree.
- Do not manually rename files; always use `IFileStorage` APIs so keys stay consistent.
