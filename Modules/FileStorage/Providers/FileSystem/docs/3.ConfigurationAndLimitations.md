# 3. Configuration and limitations

Provider-specific settings, shared guardrails, and capability boundaries.

- **Package:** `RaccoonLand.Modules.FileStorage.FileSystem`
- **Source:** `src/Configuration/FileSystemStorageOptions.cs`

## FileSystemStorageOptions

```csharp
public sealed class FileSystemStorageOptions
{
    public const string SectionName = "FileStorage:FileSystem";

    public required string RootPath { get; set; }
}
```

| Property | Required | Description |
|----------|----------|-------------|
| `RootPath` | Yes | Absolute or relative root directory for all stored objects. Created on first write per shard. |

Validation at startup: `RootPath` must not be null or whitespace.

## Shared FileStorageOptions

Bound from the `FileStorage` section — see [FileStorageOptions](../../Abstractions/docs/05.FileStorageOptions.md).

| Property | Effect on FileSystem |
|----------|---------------------|
| `MaxUploadBytes` | Rejected after streaming to temp file if length exceeds limit |
| `AllowedContentTypes` | Global MIME ceiling enforced in `ValidatePutRequest` |
| `MaxMetadataEntries` | Sidecar `userMetadata` entry count |
| `MaxMetadataValueLength` | Sidecar metadata value length |

Per-endpoint narrowing uses [FilePutConstraints](../../Abstractions/docs/11.FilePutConstraints.md) via `FileStoragePutHelper`.

## Supported IFileStorage operations

| Operation | Supported | Notes |
|-----------|-----------|-------|
| `PutAsync` | Yes | `CreateOnly`, `Overwrite`, `Upsert` via [PutMode](../../Abstractions/docs/07.PutMode.md) |
| `OpenReadAsync` | Yes | Streaming read |
| `DeleteAsync` | Yes | Removes content + sidecar |
| `ExistsAsync` | Yes | Checks content file only |
| `GetMetadataAsync` | Yes | Sidecar or filesystem fallback |

## Not supported

| Capability | Alternative |
|------------|-------------|
| `IFileUrlSigner` (presigned URLs) | Proxy through `PutAsync` / `OpenReadAsync`, or use S3 |
| `IMultipartUploadCoordinator` | Use S3 for large direct uploads |
| Cross-machine replication | External sync / shared storage / S3 |
| Built-in encryption at rest | OS-level disk encryption or encrypted volume |

Calling unsupported interfaces throws [FileStorageNotSupportedException](../../Abstractions/docs/08.Exceptions.md) from other providers; FileSystem simply does not register them.

## Deployment considerations

**Single instance.** Simplest fit — one API host writes and reads the same `RootPath`.

**Multiple instances.** All hosts must share the same `RootPath` (NFS, SMB, cloud file share). Concurrent writes to the same key are last-writer-wins; design uploads with unique keys (`PutMode.CreateOnly`).

**Containers.** Mount a persistent volume at `RootPath`. Ephemeral container filesystem loses data on restart.

**Permissions.** Run the app pool / service account with least privilege on `RootPath` only.

## Consistency model

Object files and metadata sidecars are **not** a single atomic unit:

- The object file is committed first.
- On `CreateOnly`, metadata failure rolls back the object — all-or-nothing for new keys.
- On `Overwrite` / `Upsert`, metadata is best-effort — `PutAsync` succeeds even when the sidecar commit fails; the new object bytes are durable and reads use object-only fallback until metadata is refreshed.

This matches a local-dev / single-node provider: content durability matters more than sidecar strictness on replace.
