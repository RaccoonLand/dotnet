# 2. Message templates

`FileStorageMessageTemplates` holds the stable keys produced by FileStorage exception handlers.

- **Package:** `RaccoonLand.Modules.FileStorage.AspNetCore`
- **Namespace:** `RaccoonLand.Modules.FileStorage.AspNetCore`

## Surface

```csharp
public static class FileStorageMessageTemplates
{
    public const string VALIDATION_FAILED = "FILE_STORAGE_VALIDATION_FAILED";
    public const string FILE_NOT_FOUND = "FILE_STORAGE_FILE_NOT_FOUND";
    public const string FILE_ALREADY_EXISTS = "FILE_STORAGE_FILE_ALREADY_EXISTS";
    public const string ACCESS_DENIED = "FILE_STORAGE_ACCESS_DENIED";
    public const string UNAVAILABLE = "FILE_STORAGE_UNAVAILABLE";
    public const string NOT_SUPPORTED = "FILE_STORAGE_NOT_SUPPORTED";
    public const string CONFIGURATION_ERROR = "FILE_STORAGE_CONFIGURATION_ERROR";
    public const string OPERATION_FAILED = "FILE_STORAGE_OPERATION_FAILED";
}
```

| Key | Status |
| --- | --- |
| `FILE_STORAGE_VALIDATION_FAILED` | 400 |
| `FILE_STORAGE_FILE_NOT_FOUND` | 404 |
| `FILE_STORAGE_FILE_ALREADY_EXISTS` | 409 |
| `FILE_STORAGE_ACCESS_DENIED` | 403 |
| `FILE_STORAGE_UNAVAILABLE` | 503 |
| `FILE_STORAGE_NOT_SUPPORTED` | 501 |
| `FILE_STORAGE_CONFIGURATION_ERROR` | 500 |
| `FILE_STORAGE_OPERATION_FAILED` | 500 |

## Why AspNetCore (not Abstractions)

These keys are an HTTP / pipeline presentation concern: they are emitted only by exception mapping in this package. Storage Abstractions stays free of localization. Apps that register `AddRaccoonLandFileStorageExceptionHandling` already reference AspNetCore and can use these constants when seeding translations.

## Convention

Keys use a `FILE_STORAGE_` prefix so they do not collide with application templates such as `ACCESS_DENIED` or `OPERATION_FAILED`. Register translations for these keys in your localization store.

User-facing messages have **no parameters**. Storage keys stay on the exception for logging; they are not placed in `PipelineMessage.Message`.
