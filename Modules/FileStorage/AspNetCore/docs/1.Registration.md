# 1. Registration

ASP.NET Core adapter for FileStorage: form-file mapping, streaming file results, and exception handling.

- **Package:** `RaccoonLand.Modules.FileStorage.AspNetCore`
- **Namespace:** `RaccoonLand.Modules.FileStorage.AspNetCore`
- **Depends on:** `RaccoonLand.Modules.FileStorage.Abstractions`, `RaccoonLand.Core.Hosting.AspNetCore`, `RaccoonLand.Modules.Middlewares.ExceptionHandlingMiddleware`, `RaccoonLand.Modules.MessageLocalization.Abstraction`

## Install

```bash
dotnet add package RaccoonLand.Modules.FileStorage.AspNetCore
```

Also reference a provider package (for example FileSystem) and register it as usual.

## Exception handling

Pick **one** surface (samples use HTTP):

```csharp
services.AddRaccoonLandHttpExceptionHandling(options =>
{
    options.AddFileStorageHandlers();
    // … app handlers
});
```

Or the request-pipeline middleware:

```csharp
services.AddRaccoonLandExceptionHandling(options =>
{
    options.AddFileStorageHandlers();
    // … app handlers
});
```

`AddRaccoonLandFileStorageExceptionHandling()` registers both surfaces via `PostConfigure` if you want that.

Handlers cover:

| Exception | Status | Message template key |
|-----------|--------|----------------------|
| `FileStorageValidationException` | 400 | `FILE_STORAGE_VALIDATION_FAILED` |
| `FileNotFoundStorageException` | 404 | `FILE_STORAGE_FILE_NOT_FOUND` |
| `FileAlreadyExistsStorageException` | 409 | `FILE_STORAGE_FILE_ALREADY_EXISTS` |
| `FileAccessDeniedStorageException` | 403 | `FILE_STORAGE_ACCESS_DENIED` |
| `FileStorageUnavailableException` | 503 | `FILE_STORAGE_UNAVAILABLE` |
| `FileStorageNotSupportedException` | 501 | `FILE_STORAGE_NOT_SUPPORTED` |
| `FileStorageConfigurationException` | 500 | `FILE_STORAGE_CONFIGURATION_ERROR` |
| Other `FileStorageException` | 500 | `FILE_STORAGE_OPERATION_FAILED` |

Keys live in `FileStorageMessageTemplates` (this package). See [Message templates](2.MessageTemplates.md). Handlers resolve text via `IMessageLocalization` when registered; otherwise the template key is returned as the message. Storage keys are never included in the response body.

## Form file → upload DTO

```csharp
await using var upload = file.ToFileUploadContent();

var put = await fileStorage.PutAsync(
    FileStoragePutHelper.CreateRequest(
        upload.Content,
        upload.ContentType,
        FilePutConstraints.Images,
        contentLength: upload.ContentLength),
    ct);
```

`FileUploadContent` lives in Abstractions (web-agnostic). Dispose it when finished (or use `await using`).

## OpenRead → file result

```csharp
// Inline (browser video / image) with Range support
return await fileStorage.OpenReadActionResultAsync(key, cancellationToken: ct);

// Attachment download
return await fileStorage.OpenReadFileResultAsync(
    key,
    downloadFileName: "resume.pdf",
    cancellationToken: ct);
```

Stream ownership transfers to the `FileStreamResult`. Do not dispose the open stream in the action.

## Notes

- This package does **not** register a storage provider.
- Authorize before `OpenRead` / put in application code; exception mapping alone must not leak key existence past your auth checks.
