# 6. Inbox (IInboxStore)

Consumer-side idempotency gate for service events.

- **Package:** `RaccoonLand.Modules.Messaging.Abstractions`
- **Namespace:** `RaccoonLand.Modules.Messaging.Abstractions`

## Why

The outbox relay + RabbitMQ deliver **at-least-once**. Cross-service handlers must not apply the same
`EventId` twice. The inbox records claims/completions per consuming service database.

## Surface

```csharp
public interface IInboxStore
{
    Task<InboxClaimAttempt> TryClaimAsync(
        Guid eventId,
        string eventType,
        TimeSpan claimLease,
        CancellationToken cancellationToken = default);

    Task MarkProcessedAsync(InboxClaimToken claim, CancellationToken cancellationToken = default);

    Task ReleaseAsync(
        InboxClaimToken claim,
        bool clearClaimImmediately = true,
        CancellationToken cancellationToken = default);
}

public readonly record struct InboxClaimToken(Guid EventId, DateTimeOffset ClaimedOnUtc);

public sealed class InboxClaimAttempt
{
    public InboxClaimResult Result { get; init; }
    public InboxClaimToken? Token { get; init; } // required when Result == Claimed
}

public enum InboxClaimResult
{
    Claimed,
    AlreadyProcessed,
    ClaimHeldByOther,
}
```

| Result | Consumer action |
| --- | --- |
| `Claimed` | Run handlers → `MarkProcessedAsync(token)` → ACK |
| `AlreadyProcessed` | ACK and skip |
| `ClaimHeldByOther` | NACK + requeue |

`MarkProcessedAsync` / `ReleaseAsync` require the fencing token from a successful claim so a stale worker cannot complete or clear another worker’s live claim after lease reclaim.

On handler failure call `ReleaseAsync(token)` then NACK (requeue configurable).
