# 1. RabbitMqServiceEventPublisher

RabbitMQ implementation of [`IServiceEventPublisher`](../Abstractions/docs/4.IServiceEventPublisher.md).

- **Package:** `RaccoonLand.Modules.Messaging.RabbitMQ`
- **Namespace:** `RaccoonLand.Modules.Messaging.RabbitMQ`

## Behaviour

1. Accepts only `Category = Service` outbox rows (throws otherwise).
2. Ensures a recovered RabbitMQ connection/channel (singleton publisher).
3. Optionally declares the configured exchange (`DeclareExchange`).
4. Publishes the outbox JSON `Payload` as the AMQP body (`content-type: application/json`).
5. Sets `message-id` to the event `EventId` for consumer idempotency.
6. Sets application headers listed in [ServiceEventMessageHeaders](2.ServiceEventMessageHeaders.md).
7. Uses routing key from `RoutingKeyFormat` (default `{EventType}`).
8. Publishes with **`mandatory: true`** on a channel with **publisher confirmations + confirmation tracking** enabled. If no queue is bound for the routing key (or the broker nacks), `BasicPublishAsync` throws — the outbox relay **does not** mark the row processed and will retry after the claim lease.

On any publish failure the channel/connection are reset and the exception propagates.

## Topology (default)

| Setting | Default |
| --- | --- |
| Exchange | `raccoonland.service-events` |
| Type | `topic` |
| Routing key | `EventType` (for example `order.shipped`) |
| Delivery mode | Persistent |

Consumers **must** bind a queue to the exchange with a matching pattern (for example `order.#` or `#`). An unbound routing key is a hard publish failure by design (outbox safety).

See also: [Options](3.Options.md), [Registration](4.Registration.md), [Consumer](5.Consumer.md).
