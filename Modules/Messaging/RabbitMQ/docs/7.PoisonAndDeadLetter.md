# 7. Poison / dead-letter

Controlled retries and dead-lettering for the RabbitMQ service-event consumer.

- **Package:** `RaccoonLand.Modules.Messaging.RabbitMQ`

## Policy

| Setting | Default | Behaviour |
| --- | --- | --- |
| `MaxDeliveryAttempts` | `5` | Handler failures republish to the same queue with `raccoonland-delivery-attempt` incremented. When `> 0`, a DLX **must** be configured (validated on start). |
| `EnableDeadLetterTopology` | `true` | Declares `{QueueName}.dlx` (**fanout**) + `{QueueName}.poison` and sets `x-dead-letter-exchange` on the main queue. |
| `DeadLetterRoutingKey` | `poison` | Passed on poison publish; fanout ignores routing key for delivery to bound poison queue. |
| `RequeueOnFailure` | `true` | Used only when `MaxDeliveryAttempts = 0` (legacy simple NACK). |
| `ClaimHeldByOtherRequeueDelay` | `00:00:02` | **Backoff before** NACK+requeue when another worker holds the inbox claim. Delivery stays unacked during the delay — keep `PrefetchCount` modest under contention. |

## Inbox contention vs `MaxDeliveryAttempts`

`ClaimHeldByOther` is **not** a handler failure and **does not** increment `raccoonland-delivery-attempt`
or consume `MaxDeliveryAttempts`. The message is delayed by `ClaimHeldByOtherRequeueDelay`, then
requeued so another delivery can try after the live claim lease expires or completes.

Poison budget applies only after a successful inbox claim when dispatch/processing throws.

## Flow on handler failure

1. Inbox claim is **released** (so a later attempt can reclaim).
2. If `attempt < MaxDeliveryAttempts` → republish to the queue with **publisher confirms** + `mandatory: true`, then **ACK** original. If republish fails, **NACK requeue** the original (no silent drop).
3. If attempts exhausted → publish to DLX with confirms + `mandatory: true`, then **ACK**. Publish failure → **NACK requeue**.
4. Crash between a **confirmed** republish/poison publish and ACK can still **duplicate** (at-least-once; **Inbox + idempotent handlers required**). Docs previously warned about duplicates; confirms close the **loss** window where ACK happened without a successful republish.

## Attempt counting

`raccoonland-delivery-attempt` is owned by this consumer’s republish path after claimed processing failures. Broker-native redelivery flags are **not** used for poison decisions. Inbox contention requeues do not change the header.

## Ops notes

- Inspect `{QueueName}.poison` for failed payloads.
- Inbox is **not** marked processed on poison; after fixing data you may replay from the poison queue (same `EventId` can be claimed again).
- Changing DLX arguments on an existing queue requires deleting/recreating the queue in RabbitMQ.
- Ensure publisher routing keys match consumer `BindingKeys` — unroutable publishes fail with publisher confirms (`mandatory: true`).
- Register `AddRaccoonLandInboxStore` in every consuming service; without inbox uniqueness on `EventId`, at-least-once duplicates are unsafe.
