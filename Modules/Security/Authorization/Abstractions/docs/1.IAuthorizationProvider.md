# 1. IAuthorizationProvider

`IAuthorizationProvider` is the single contract the authorization pipeline depends on. It decides whether the
current caller may execute a request, identified by the concrete request type's full name.

- **Package:** `RaccoonLand.Modules.Security.Authorization.Abstractions`
- **Namespace:** `RaccoonLand.Modules.Security.Authorization.Abstractions`

## Surface

```csharp
public interface IAuthorizationProvider
{
    Task<AuthorizationDecision> AuthorizeAsync(
        AuthorizationContext context,
        CancellationToken cancellationToken);
}
```

## Design principles

1. **Externally supplied rules.** Access rules — including whether a request is publicly accessible
   (anonymous) — come from outside the request: configuration, a database, or a remote service. The request
   carries **no** authorization metadata: no attribute and no marker interface.
2. **Request identity is the key.** Requests are identified by the **concrete** request type's
   `GetType().FullName`. The host pipeline must dispatch that concrete instance. Proxies, decorators, or
   unexpected runtime types are not supported as keys and will not match registered rules (fail-closed).
   See [AuthorizationContext](2.AuthorizationContext.md).
3. **The provider owns anonymity.** Whether an endpoint is anonymous is decided **inside** the provider, not
   in the middleware. Keeping a separate anonymous list in the middleware would create a second bypass path
   and weaken security; the provider is the single source of truth.
4. **Deny-by-default.** A request that is not explicitly anonymous or explicitly permitted must return a
   non-allowed decision.
5. **Decisions vs infrastructure failures.** `Denied` / `Unauthenticated` are authorization outcomes.
   Timeouts, transport errors, malformed payloads, and database failures must **propagate as exceptions** and
   are not silently mapped to deny. Host exception handling owns those failures. Simple in-memory providers
   may ignore `CancellationToken`.

## Single active provider

Exactly one `IAuthorizationProvider` is active at a time. The middleware resolves it per request from
`PipelineContext.RequestServices`. Choose an implementation by registering one provider package (for example
the claim-based provider).
