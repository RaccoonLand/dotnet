# 4. Registration

Register the claim-based authorization provider and its claims accessor — configuring options in code, from
`IConfiguration`, or both.

- **Package:** `RaccoonLand.Modules.Security.Authorization.Claims`
- **Namespace:** `Microsoft.Extensions.DependencyInjection` (so `AddRaccoonLandClaimAuthorization` is available wherever DI is configured)

## Surface

```csharp
public static IServiceCollection AddRaccoonLandClaimAuthorization(
    this IServiceCollection services,
    Action<ClaimAuthorizationOptions>? configure = null);

public static IServiceCollection AddRaccoonLandClaimAuthorization(
    this IServiceCollection services,
    IConfiguration configuration,
    Action<ClaimAuthorizationOptions>? configure = null);
```

Both overloads register:

| Service | Lifetime | Notes |
| --- | --- | --- |
| `IHttpContextAccessor` | singleton | via `AddHttpContextAccessor()` |
| `IClaimsPrincipalAccessor` → `HttpContextClaimsPrincipalAccessor` | singleton | `TryAddSingleton` (override by registering your own first) |
| `ClaimAuthorizationOptions` | options | configured from `configuration` and/or `configure` |
| `IAuthorizationProvider` → `ClaimAuthorizationProvider` | scoped | `TryAddScoped` |

When both a configuration and a `configure` delegate are supplied, the configuration is applied **first** and
the delegate runs **after**, so code can layer additional rules — including custom assertions — on top of
configuration.

## Code-only

```csharp
services.AddRaccoonLandAuthorization();            // middleware package
services.AddRaccoonLandClaimAuthorization(options =>
{
    options.AllowAnonymous("MyApp.Auth.Commands.LoginCommand");
    options.RequireClaim("MyApp.People.Commands.CreatePersonCommand", "permission", "people.write");
});
```

## From configuration

```csharp
services.AddRaccoonLandClaimAuthorization(
    configuration.GetSection("ClaimAuthorization"),
    options =>
    {
        // Custom assertions are code-only — they cannot come from configuration.
        options.RequireAssertion(
            "MyApp.People.Commands.DeletePersonCommand",
            principal => principal.HasClaim("department", "HR"));
    });
```

The bound section matches `ClaimAuthorizationConfiguration`:

```json
{
  "ClaimAuthorization": {
    "AnonymousRequests": [
      "MyApp.Auth.Commands.LoginCommand"
    ],
    "Policies": {
      "MyApp.People.Queries.GetMyProfileQuery": {
        "Claims": []
      },
      "MyApp.People.Commands.CreatePersonCommand": {
        "Claims": [
          { "ClaimType": "permission", "AllowedValues": [ "people.write" ] }
        ]
      },
      "MyApp.People.Commands.DeletePersonCommand": {
        "Claims": [
          { "ClaimType": "role", "AllowedValues": [ "admin", "owner" ] }
        ]
      }
    }
  }
}
```

- A policy with an empty `Claims` array means "any authenticated caller".
- `AllowedValues` is an OR set; an empty array accepts any value of `ClaimType`.

## Non-HTTP hosts

`HttpContextClaimsPrincipalAccessor` depends on `IHttpContextAccessor`. In a worker or other non-HTTP host,
register a custom `IClaimsPrincipalAccessor` **before** calling this method so the `TryAddSingleton` keeps your
implementation.

See also: [ClaimAuthorizationProvider](3.ClaimAuthorizationProvider.md), [ClaimAuthorizationOptions](2.ClaimAuthorizationOptions.md).
