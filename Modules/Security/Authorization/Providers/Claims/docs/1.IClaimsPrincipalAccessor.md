# 1. IClaimsPrincipalAccessor

`IClaimsPrincipalAccessor` supplies the current caller's `ClaimsPrincipal` to the claim-based authorization
provider.

- **Package:** `RaccoonLand.Modules.Security.Authorization.Claims`
- **Namespace:** `RaccoonLand.Modules.Security.Authorization.Claims.Principals`

## Surface

```csharp
public interface IClaimsPrincipalAccessor
{
    ClaimsPrincipal? Principal { get; }
}
```

| Member | Description |
| --- | --- |
| `Principal` | The current principal, or `null` when none is available. |

## Why it lives here (not in Abstractions)

Only the claim-based provider needs a `ClaimsPrincipal`. Other providers (database- or API-backed) identify
the caller through `ICurrentExecutionContext.UserId` instead. Keeping this contract in the provider package
avoids pushing a claims dependency onto the shared abstractions.

`System.Security.Claims.ClaimsPrincipal` is part of the base class library, so the contract itself does not
depend on ASP.NET Core.

## Default implementation

`HttpContextClaimsPrincipalAccessor` returns `HttpContext.User` from the ambient `IHttpContextAccessor`:

```csharp
public sealed class HttpContextClaimsPrincipalAccessor(IHttpContextAccessor httpContextAccessor)
    : IClaimsPrincipalAccessor
{
    public ClaimsPrincipal? Principal => httpContextAccessor.HttpContext?.User;
}
```

It is registered automatically by [`AddRaccoonLandClaimAuthorization`](4.Registration.md) (via `TryAddSingleton`,
so a custom accessor registered earlier wins). In a non-HTTP host (for example a worker), register your own
implementation that returns the principal for the current execution scope.
