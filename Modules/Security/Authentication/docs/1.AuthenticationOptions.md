# 1. AuthenticationOptions

`AuthenticationOptions` is the root configuration model bound from the `Authentication` section. It defines
default schemes, claim-mapping behaviour, and dictionaries of JWT Bearer and OpenID Connect schemes.

- **Package:** `RaccoonLand.Modules.Security.Authentication`
- **Namespace:** `RaccoonLand.Modules.Security.Authentication.Configuration`

## Surface

```csharp
public sealed class AuthenticationOptions
{
    public const string SectionName = "Authentication";

    public string? DefaultScheme { get; set; }
    public string? DefaultAuthenticateScheme { get; set; }
    public string? DefaultChallengeScheme { get; set; }
    public string? DefaultSignInScheme { get; set; }
    public string? DefaultSignOutScheme { get; set; }

    public bool DisableDefaultClaimMapping { get; set; } = true;
    public bool ClearGlobalJwtClaimTypeMaps { get; set; }
    public bool AllowExternalDefaultSchemes { get; set; }

    public Dictionary<string, JwtBearerOptions> JwtBearer { get; set; }
    public Dictionary<string, OpenIdConnectOptions> OpenIdConnect { get; set; }
}
```

| Member | Description |
| --- | --- |
| `DefaultScheme` | Default authentication scheme for ASP.NET Core `AuthenticationOptions`. |
| `DefaultAuthenticateScheme` | Scheme used to populate `HttpContext.User`. |
| `DefaultChallengeScheme` | Scheme used when an endpoint requires authentication. |
| `DefaultSignInScheme` / `DefaultSignOutScheme` | Schemes for sign-in and sign-out. |
| `DisableDefaultClaimMapping` | When `true` (default), sets `MapInboundClaims = false` on every scheme registered by this package so claim types stay provider-native (`sub`, `email`, …). Does **not** clear process-wide static maps. |
| `ClearGlobalJwtClaimTypeMaps` | When `true` (default `false`), clears Microsoft's process-wide JWT claim type maps. Opt-in only — irreversible for the process lifetime and affects other JWT handlers in the process. Prefer `DisableDefaultClaimMapping` for scheme-local behaviour. |
| `AllowExternalDefaultSchemes` | When `false` (default), each set `Default*` must name a JwtBearer or OpenIdConnect scheme from this registration. When `true`, defaults may name schemes registered elsewhere (for example Cookie). **Only the name is accepted** — DI is not checked for a real scheme; a typo surfaces when authentication runs. Empty/whitespace defaults are always rejected. |
| `JwtBearer` | Named JWT Bearer schemes — see [SchemeOptions](2.SchemeOptions.md). Must not be `null`; entries must not be `null`. Keys use `StringComparer.OrdinalIgnoreCase` (package policy). |
| `OpenIdConnect` | Named OpenID Connect schemes — see [SchemeOptions](2.SchemeOptions.md). Must not be `null`; entries must not be `null`. Keys use `StringComparer.OrdinalIgnoreCase` (package policy). |

## Scheme name comparison

This package **deliberately** treats scheme names as case-insensitive for uniqueness and JWT/OIDC collision
checks, to reject ambiguous names (`Bearer` and `bearer` collide). That is a package rule, not a description of
ASP.NET Core's `AuthenticationSchemeProvider`, which stores and matches scheme names with ordinal /
case-sensitive comparison and can treat `Bearer` and `bearer` as two different schemes.

## Defaults and external schemes

By default (`AllowExternalDefaultSchemes = false`), package defaults must name a scheme listed under `JwtBearer`
or `OpenIdConnect` for this registration. This catches typos in API/JWT apps at startup.

For interactive **Cookie + OpenID Connect**, set `AllowExternalDefaultSchemes` to `true` so defaults such as
`DefaultScheme` / `DefaultSignInScheme` can point at a Cookie scheme registered by the application
(this package does not call `AddCookie`). The package does **not** verify that the external scheme exists in DI;
a misspelled Cookie scheme name fails later when authentication is used.

## Example (API / JWT)

```json
{
  "Authentication": {
    "DefaultScheme": "Bearer",
    "DefaultAuthenticateScheme": "Bearer",
    "DefaultChallengeScheme": "Bearer",
    "DisableDefaultClaimMapping": true,
    "ClearGlobalJwtClaimTypeMaps": false,
    "AllowExternalDefaultSchemes": false,
    "JwtBearer": {
      "Bearer": {
        "Authority": "https://login.example.com",
        "Audience": "my-api"
      }
    }
  }
}
```

## Example (Cookie + OIDC)

```json
{
  "Authentication": {
    "DefaultScheme": "Cookies",
    "DefaultChallengeScheme": "oidc",
    "DefaultSignInScheme": "Cookies",
    "AllowExternalDefaultSchemes": true,
    "OpenIdConnect": {
      "oidc": {
        "Authority": "https://login.example.com",
        "ClientId": "web-client",
        "ResponseType": "code"
      }
    }
  }
}
```

The application still registers Cookie separately, for example `builder.Services.AddAuthentication().AddCookie("Cookies")`
before or alongside this package's registration as appropriate for the host.
