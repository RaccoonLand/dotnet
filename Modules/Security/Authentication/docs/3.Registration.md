# 3. Registration (AddRaccoonLandAuthentication)

`AddRaccoonLandAuthentication` registers JWT Bearer and OpenID Connect schemes from configuration. It is
**provider-agnostic** — any standards-compliant IdP works as long as the scheme options are set correctly.

- **Package:** `RaccoonLand.Modules.Security.Authentication`
- **Namespace:** `RaccoonLand.Modules.Security.Authentication`

## Surface

```csharp
public static IServiceCollection AddRaccoonLandAuthentication(
    this IServiceCollection services,
    IConfiguration configuration,
    string sectionName = AuthenticationOptions.SectionName,
    Action<AuthenticationOptions>? configureOptions = null);
```

| Parameter | Description |
| --- | --- |
| `configuration` | Application configuration (`appsettings`, env vars, user secrets, Key Vault, …). |
| `sectionName` | Root section (default `Authentication`). Must be non-empty. |
| `configureOptions` | Optional code-based override after binding, before validation and registration. |

## Binding precedence

Settings are finalized on a **single path** used for both DI and scheme registration:

1. Root scalar properties are bound once from the root section (scheme dictionaries are excluded from this step).
2. Each `JwtBearer` / `OpenIdConnect` child section is bound once via Merge into the scheme dictionaries.
3. `configureOptions` is applied as a code-based override.
4. Options are validated (null dictionaries/entries, scheme names, case-insensitive collisions, defaults).
5. Authentication schemes are registered (`AddAuthentication`, `AddJwtBearer`, `AddOpenIdConnect`).
6. Optional process-wide claim-map clearing when `ClearGlobalJwtClaimTypeMaps` is `true`.
7. The finalized snapshot is registered as `IOptions<AuthenticationOptions>`, then the one-call marker.

`IOptions` and the marker are registered **last** so a failure during scheme registration does not mark the call as
completed via the marker. Absence of the marker after a failure does **not** mean retrying on the same
`IServiceCollection` is safe: `IServiceCollection` is not transactional, and partial scheme descriptors from the
failed attempt may remain and duplicate on retry. Prefer a fresh service collection (or process restart) after a
failed registration.

Handler options registered by ASP.NET Core do not reload when configuration files change after startup;
treat authentication registration as a one-time startup snapshot.

## What it registers

1. `AddAuthentication` with default schemes from [`AuthenticationOptions`](1.AuthenticationOptions.md).
2. One `AddJwtBearer` per entry in `JwtBearer`.
3. One `AddOpenIdConnect` per entry in `OpenIdConnect`.
4. Scheme-local `MapInboundClaims = false` when `DisableDefaultClaimMapping` is `true`.
5. Optional process-wide claim-map clearing when `ClearGlobalJwtClaimTypeMaps` is `true`
   (irreversible for the process lifetime; affects other JWT handlers in the process).
6. `IOptions<AuthenticationOptions>` — the finalized startup snapshot (registered after schemes).

This package does **not** register Cookie (or other external) schemes. For Cookie + OIDC, register Cookie in the
application and set [`AllowExternalDefaultSchemes`](1.AuthenticationOptions.md) to `true`.

## Multiple calls

Calling `AddRaccoonLandAuthentication` more than once on the same `IServiceCollection` is **not supported**
and throws `InvalidOperationException`. Register all schemes in a single call (configuration and/or
`configureOptions`).

## Validation (fail-fast)

Startup fails with `InvalidOperationException` when:

- `JwtBearer` or `OpenIdConnect` is `null` after `configureOptions` (use an empty dictionary instead).
- A dictionary entry value is `null` (for example `options.JwtBearer["Bearer"] = null`).
- A scheme name under `JwtBearer` or `OpenIdConnect` is null, empty, or whitespace.
- The same scheme name appears twice under one dictionary or under both `JwtBearer` and `OpenIdConnect`
  (case-insensitive package uniqueness rule).
- A default scheme property is empty/whitespace.
- A default scheme names a scheme not registered by this call, unless `AllowExternalDefaultSchemes` is `true`
  (external names are not checked for real DI registration).
- This method was already called on the service collection.

## What it does **not** register

The application remains responsible for:

- `app.UseAuthentication()` and `app.UseAuthorization()`
- Authorization policies (`AddAuthorization`, `[Authorize]`, …)
- Swagger / OpenAPI security definitions
- Cookie (or other) schemes when using interactive OIDC

This keeps authentication wiring separate from authorization and API documentation.

## Example

```csharp
using RaccoonLand.Modules.Security.Authentication;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddRaccoonLandAuthentication(builder.Configuration);

builder.Services.AddAuthorization();
// builder.Services.AddAuthorizationBuilder() ...

var app = builder.Build();

app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();
app.Run();
```
