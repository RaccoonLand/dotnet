# 1. FluentValidationMiddleware

`FluentValidationMiddleware` is a class-based [`IPipelineMiddleware`](../../../Core/RequestProcessing/RequestProcessing.Abstractions/docs/4.PipelineDelegate.md)
that runs FluentValidation validators for the current request type before the terminal endpoint executes.

- **Package:** `RaccoonLand.Modules.Middlewares.FluentValidationMiddleware`
- **Namespace:** `RaccoonLand.Modules.Middlewares.FluentValidationMiddleware`

## Host independence

This middleware has **no dependency on ASP.NET Core**. It does not set HTTP status codes — it only assigns
`context.Response` with validation errors. A host adapter maps that envelope to transport-specific output
(for example HTTP 400 when `Errors` is non-empty).

## Behaviour

1. Resolve all `IValidator<TRequest>` implementations for `context.Request` from
   **`PipelineContext.RequestServices`** (validators are typically registered as scoped or transient).
2. If none exist, call `next(context)` and return.
3. Run each validator against the request; collect failures.
4. If there are no failures, call `next(context)`.
5. Otherwise **short-circuit**: set `context.Response` to a [`PipelineResponse`](../../../Core/RequestProcessing/RequestProcessing.Abstractions/docs/7.PipelineResponse.md)
   with one [`PipelineMessage`](../../../Core/RequestProcessing/RequestProcessing.Abstractions/docs/7.PipelineResponse.md)
   per failure and **do not** call `next`.

## Error shape

| Field | Source |
| --- | --- |
| `Code` | `ValidationFailure.ErrorCode` when non-empty; otherwise `ValidationFailure.PropertyName`. |
| `Message` | `ValidationFailure.ErrorMessage` as returned by the validator. **Localization is the validator author's responsibility** — inject `IMessageLocalization` into `AbstractValidator<T>` and resolve message keys there. |

This matches the error envelope produced by
[`ExceptionHandlingMiddleware`](../ExceptionHandlingMiddleware/docs/1.ExceptionHandlingMiddleware.md) for
domain failures — consumers see a consistent `PipelineResponse.Errors` list.

## Middleware contract

The middleware is registered as a **singleton** and is **stateless**:

- `ILogger<FluentValidationMiddleware>` is injected via the constructor (singleton-safe).
- Validators are resolved from `context.RequestServices` inside `InvokeAsync`.

See [PipelineDelegate & IPipelineMiddleware](../../../Core/RequestProcessing/RequestProcessing.Abstractions/docs/4.PipelineDelegate.md).

## Pipeline placement

Add **inside** exception-handling middleware (if present) and **before** the terminal endpoint:

```csharp
pipeline
    .UseMiddleware<ExceptionHandlingMiddleware>()
    .UseMiddleware<FluentValidationMiddleware>();
```

Validation failures are expected outcomes — they short-circuit without throwing, so exception-handling
middleware does not need to catch them.

See also: [Registration](2.Registration.md).
