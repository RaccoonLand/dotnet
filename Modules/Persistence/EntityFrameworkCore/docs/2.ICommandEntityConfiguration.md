# 2. ICommandEntityConfiguration

Marker interface for entity type configurations that belong on the **command** (write) `DbContext`.

- **Package:** `RaccoonLand.Modules.Persistence.EntityFrameworkCore`
- **Namespace:** `RaccoonLand.Modules.Persistence.EntityFrameworkCore.Configuration`

## Surface

```csharp
public interface ICommandEntityConfiguration;
```

The interface carries **no members**. It exists only to tag configuration classes at compile time so
[`ApplyConfigurationsFromAssembly<ICommandEntityConfiguration>`](4.ApplyConfigurationsFromAssembly.md)
can discover them.

## Implementing a configuration

Implement both `IEntityTypeConfiguration<TEntity>` and `ICommandEntityConfiguration` on the same class:

```csharp
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;
using RaccoonLand.Modules.Persistence.EntityFrameworkCore.Configuration;

namespace MyApp.People.Persistence.Commands;

public sealed class PersonConfiguration : IEntityTypeConfiguration<Person>, ICommandEntityConfiguration
{
    public void Configure(EntityTypeBuilder<Person> builder)
    {
        builder.ToTable("People");
        builder.HasKey(x => x.Id);
    }
}
```

## Rules

1. Place command configurations under the command persistence folder for the capability.
2. **Do not** also implement `IQueryEntityConfiguration` on the same class. Dual markers are **forbidden** and
   cause `InvalidOperationException` during discovery.
3. Implement **exactly one** closed `IEntityTypeConfiguration<TEntity>` per class (see
   [ApplyConfigurationsFromAssembly](4.ApplyConfigurationsFromAssembly.md)).
4. Keep one configuration class per aggregate (or read model entity) unless a deliberate split is needed.

See also: [Overview](1.Overview.md), [ApplyConfigurationsFromAssembly](4.ApplyConfigurationsFromAssembly.md).
