# 3. IQueryEntityConfiguration

Marker interface for entity type configurations that belong on the **query** (read) `DbContext`.

- **Package:** `RaccoonLand.Modules.Persistence.EntityFrameworkCore`
- **Namespace:** `RaccoonLand.Modules.Persistence.EntityFrameworkCore.Configuration`

## Surface

```csharp
public interface IQueryEntityConfiguration;
```

The interface carries **no members**. It exists only to tag configuration classes at compile time so
[`ApplyConfigurationsFromAssembly<IQueryEntityConfiguration>`](4.ApplyConfigurationsFromAssembly.md)
can discover them.

## Implementing a configuration

Implement both `IEntityTypeConfiguration<TEntity>` and `IQueryEntityConfiguration` on the same class:

```csharp
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;
using RaccoonLand.Modules.Persistence.EntityFrameworkCore.Configuration;

namespace MyApp.People.Persistence.Queries;

public sealed class PersonReadConfiguration
    : IEntityTypeConfiguration<PersonRead>,
      IQueryEntityConfiguration
{
    public void Configure(EntityTypeBuilder<PersonRead> builder)
    {
        builder.ToView("vw_People");
        builder.HasNoKey();
    }
}
```

## Rules

1. Place query configurations under the query persistence folder for the capability.
2. Read models may differ from command aggregates (denormalized views, keyless types, etc.) — the marker
   only controls **which DbContext** picks up the configuration, not the mapping shape.
3. Do **not** add `IQueryEntityConfiguration` to command-side configurations.

See also: [Overview](1.Overview.md), [ApplyConfigurationsFromAssembly](4.ApplyConfigurationsFromAssembly.md).
