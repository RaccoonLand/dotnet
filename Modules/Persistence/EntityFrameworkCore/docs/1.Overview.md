# 1. Overview

Shared Entity Framework Core primitives for RaccoonLand persistence layers. The package is
**provider-agnostic** ‚Äî it depends only on `Microsoft.EntityFrameworkCore`, not on SQL Server or any
other database provider.

- **Package:** `RaccoonLand.Modules.Persistence.EntityFrameworkCore`
- **Namespaces:**
  - `RaccoonLand.Modules.Persistence.EntityFrameworkCore.Configuration` ‚Äî marker interfaces
  - `RaccoonLand.Modules.Persistence.EntityFrameworkCore.Extensions` ‚Äî `ModelBuilder` helpers

## Build / package settings

This project's `.csproj` only declares the EF Core package reference. `TargetFramework`, nullable,
documentation/symbols, and shared package metadata come from the repo-root `Directory.Build.props`.
The EF Core version is managed by central package management (`Directory.Packages.props`).

## Problem

In a CQRS setup the command and query `DbContext` types often live in the same solution assembly (or share
an assembly). Both sides may define `IEntityTypeConfiguration<TEntity>` classes, but each `DbContext` must
apply **only its own** configurations. The built-in
`ModelBuilder.ApplyConfigurationsFromAssembly(assembly)` applies every configuration in the assembly with no
way to separate command from query.

## Solution

1. Tag each configuration class with a **marker interface**:
   - [`ICommandEntityConfiguration`](2.ICommandEntityConfiguration.md) for the write model
   - [`IQueryEntityConfiguration`](3.IQueryEntityConfiguration.md) for the read model
2. Apply configurations through the marker-aware extension
   [`ApplyConfigurationsFromAssembly<TMarker>`](4.ApplyConfigurationsFromAssembly.md).

## Typical layout

```
MyApp/
  People/
    Persistence/
      Commands/
        PersonConfiguration.cs          // IEntityTypeConfiguration<Person> + ICommandEntityConfiguration
      Queries/
        PersonReadConfiguration.cs      // IEntityTypeConfiguration<PersonRead> + IQueryEntityConfiguration
  Shared/
    Persistence/
      Commands/
        MyAppCommandDbContext.cs
      Queries/
        MyAppQueryDbContext.cs
```

Command and query database packages (for example
[`RaccoonLand.Modules.Persistence.SqlServer.Commands`](../SqlServer/Commands/docs/1.BaseCommandDbContext.md)
and
[`RaccoonLand.Modules.Persistence.SqlServer.Queries`](../SqlServer/Queries/docs/1.BaseQueryDbContext.md))
remain responsible for `DbContext` base behaviour; this package only adds the configuration discovery
convention.

See also: [ICommandEntityConfiguration](2.ICommandEntityConfiguration.md),
[IQueryEntityConfiguration](3.IQueryEntityConfiguration.md),
[ApplyConfigurationsFromAssembly](4.ApplyConfigurationsFromAssembly.md).
