# 2. IOutboxWriter

`IOutboxWriter` is the **request-scoped** collector applications use to enqueue outbox messages during command
handling. Buffered messages are flushed by the implementation when the ambient persistence unit of work
completes (typically EF Core `SaveChanges`).

- **Package:** `RaccoonLand.Modules.Persistence.Outbox.Abstraction`
- **Namespace:** `RaccoonLand.Modules.Persistence.Outbox.Abstraction`

## Surface

```csharp
public interface IOutboxWriter
{
    void Enqueue<TOutbox>(object payload, OutboxEnqueueOptions? options = null)
        where TOutbox : IOutbox;
}
```

| Parameter | Description |
| --- | --- |
| `TOutbox` | The target channel marker ([`IOutbox`](1.IOutbox.md)). Must be registered. |
| `payload` | Non-null message body; serialized to JSON into [`OutboxPendingMessage.Payload`](4.OutboxPendingMessage.md). Prefer a typed DTO. |
| `options` | Optional correlation metadata ([`OutboxEnqueueOptions`](3.OutboxEnqueueOptions.md)). Prefer an explicit `EventType`. |

## Atomicity (conditional guarantee)

Transactional **all-or-nothing** with entity changes is **not** implied by the interface alone (there is no
`DbContext` on the contract). It holds **only when**:

1. The writer buffer is flushed by the same persistence pipeline that saves the business changes, and
2. That flush uses the **same connection and transaction** (or equivalent unit of work) as those changes.

Hosts must register an implementation that meets those conditions (for example an EF Core `SaveChanges`
interceptor writing on `Database.CurrentTransaction`). If flush uses a separate connection/transaction,
atomicity is **not** guaranteed by this abstraction.

## Buffer lifecycle (multi-`SaveChanges`)

Every compliant implementation must honor:

| Step | Behaviour |
| --- | --- |
| `Enqueue` | Appends to a scoped in-memory buffer. Nothing is written to storage yet. |
| Successful flush | Messages in the buffer are persisted with the unit of work, then **removed** from the buffer. |
| Failed unit of work before a successful flush | Buffered messages **remain** so a later `SaveChanges` / retry can flush them. |
| Failed flush | Must not silently drop messages: clear the buffer only after a successful write (or restore on failure). |
| Further `Enqueue` after a successful flush | Belongs to the **next** unit of work / `SaveChanges`. |
| Multiple `SaveChanges` in one scope | Each successful flush drains what was pending at that moment; later enqueues are not re-sent with earlier saves. |

Duplicate risk: the abstraction does **not** deduplicate. If the application retries a whole command after a
commit that already flushed outbox rows, it may enqueue again â€” use message `Id` / consumer idempotency.

## Lifetime and thread-safety

- **Scoped** to one request / unit of work (typical DI: scoped).
- **Not** designed for concurrent use of the same instance.
- **Do not** use a shared singleton instance from background services unless a specific implementation
  documents a different model.
- Enqueue order within one scope is **FIFO** for that buffer.
- No built-in deduplication of identical payloads.

## Serialization (contract for `payload`)

| Rule | Detail |
| --- | --- |
| Null | `payload` must not be null (`ArgumentNullException`). |
| Result | After enqueue, [`OutboxPendingMessage.Payload`](4.OutboxPendingMessage.md) is always **valid JSON text**. |
| Serializer | Choice and options (including polymorphism) belong to the **implementation**. |
| Flush | Storage writers must **not** JSON-serialize `Payload` again â€” store the string as-is. |
| Pre-built JSON strings | Passing a `string` that is already JSON typically produces a JSON **string** value (quoted), not raw object JSON. Prefer DTOs. |

## Unregistered channel

Enqueueing for a channel that was not registered throws `InvalidOperationException`.

## Usage

```csharp
public sealed class PlaceOrderEndpoint(IOutboxWriter outbox, ShopCommandDbContext db)
    : IEndpoint<PlaceOrderCommand>
{
    public async Task<Result> ExecuteAsync(PlaceOrderCommand request, CancellationToken cancellationToken)
    {
        var order = Order.Place(request);
        db.Orders.Add(order);

        outbox.Enqueue<IQueueOutbox>(
            new OrderPlacedMessage(order.Id),
            new OutboxEnqueueOptions { EventType = "order.placed.v1" });

        await db.SaveChangesAsync(cancellationToken); // order + outbox when implementation is transactional
        return Result.Success();
    }
}
```

> Channel registration APIs such as `AddRaccoonLandOutbox` belong to concrete packages (for example SQL Server
> Commands), not to this abstraction.

See also: [OutboxEnqueueOptions](3.OutboxEnqueueOptions.md), [OutboxPendingMessage](4.OutboxPendingMessage.md).
