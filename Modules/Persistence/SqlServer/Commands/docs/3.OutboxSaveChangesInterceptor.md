# 3. OutboxSaveChangesInterceptor

`OutboxSaveChangesInterceptor` persists the **domain and service events** held by aggregate roots into an
outbox table, atomically with the aggregate changes (transactional outbox pattern).

- **Package:** `RaccoonLand.Modules.Persistence.SqlServer.Commands`
- **Namespace:** `RaccoonLand.Modules.Persistence.SqlServer.Commands.Interceptors`

## When it runs

It overrides `SavedChangesAsync` — EF Core has already written the entities, but the surrounding transaction
opened by [`BaseCommandDbContext`](1.BaseCommandDbContext.md) has **not** committed yet. The rows are written
with **Dapper** on the same `DbConnection` and `DbTransaction`, so events and entity changes commit together.

## What it writes

For every tracked `IAggregateRoot` that has pending events, one `OutboxMessage` row per event:

| Column | Source |
| --- | --- |
| `Id` | `DomainEvent.EventId` / `ServiceEvent.EventId` (idempotency). |
| `Category` | `"Domain"` or `"Service"`. |
| `EventType` | `event.EventType`. |
| `AggregateType` | CLR type name of the aggregate. |
| `AggregateBusinessKey` | `IAggregateRoot.BusinessKey`. |
| `Payload` | JSON-serialized event (web defaults). |
| `CreatedBy` | The aggregate's `ModifiedBy ?? CreatedBy` (stamped earlier by the [audit interceptor](2.AuditSaveChangesInterceptor.md)). |
| `OccurredOnUtc` | `event.OccurredOnUtc`. |
| `CreatedOnUtc` | Now (UTC). |

After writing, `ClearDomainEvents()` / `ClearServiceEvents()` are called on each aggregate.

## Why it does not filter by EntityState

Because the interceptor runs on `SavedChangesAsync` — **after** EF Core has called `AcceptAllChanges` — the
entity states are already reset (`Added`/`Modified` → `Unchanged`, `Deleted` → `Detached`). Filtering by
`EntityState.Added/Modified/Deleted` at this point would match **nothing**. The raised events, however, are
still held in memory on the aggregates (they are custom collections, untouched by `AcceptAllChanges`), so the
interceptor selects aggregates that carry events:

```csharp
context.ChangeTracker
    .Entries<IAggregateRoot>()
    .Select(entry => entry.Entity)
    .Where(aggregate => aggregate.DomainEvents.Count > 0 || aggregate.ServiceEvents.Count > 0);
```

## OutboxOptions

`OutboxOptions` controls the target table. The table may live in the current database or in another database
on the **same SQL Server instance** (three-part naming) so the write stays inside the same transaction without
DTC. Cross-instance / DTC scenarios are intentionally unsupported.

```csharp
public sealed class OutboxOptions
{
    public string? Database { get; set; } // null => current database
    public string Schema { get; set; } = "dbo";
    public string Table { get; set; } = "OutboxEvent";
    public string QualifiedTableName { get; }   // [Db].[Schema].[Table] or [Schema].[Table]
}
```

## Ordering with the audit interceptor

The audit interceptor runs on `SavingChanges` (before save) and stamps the acting user; this interceptor runs
on `SavedChanges` (after save, before commit) and reads that user onto the outbox rows. Register them in the
order shown in [Registration](4.Registration.md).

See also: [BaseCommandDbContext](1.BaseCommandDbContext.md), [Registration](4.Registration.md).
