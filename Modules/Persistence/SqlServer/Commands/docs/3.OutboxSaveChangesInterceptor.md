# 3. OutboxSaveChangesInterceptor

`OutboxSaveChangesInterceptor` persists the **domain and service events** held by aggregate roots into an
outbox table, atomically with the aggregate changes (transactional outbox pattern).

- **Package:** `RaccoonLand.Modules.Persistence.SqlServer.Commands`
- **Namespace:** `RaccoonLand.Modules.Persistence.SqlServer.Commands.Interceptors`

## When it runs

| Hook | Behaviour |
| --- | --- |
| `SavedChangesAsync` | After EF has written entities; INSERTs outbox rows with Dapper on the current connection/transaction. Does **not** clear in-memory events yet. |
| `TransactionCommitted` | Removes **only** domain/service events whose `EventId` was written in this transaction via the internal `IAggregateRootMutations` surface (`RemoveDomainEvents` / `RemoveServiceEvents`). Newer unwritten events on the same aggregate are kept. |
| `TransactionRolledBack` | Clears per-context flush marks (events stay on aggregates). |

Only the **async** save path is supported (`BaseCommandDbContext` blocks synchronous `SaveChanges`).

## What it writes

For every tracked `IAggregateRoot` that has pending events, one `OutboxMessage` row per event:

| Column | Source |
| --- | --- |
| `Id` | `DomainEvent.EventId` / `ServiceEvent.EventId` (idempotency). |
| `Category` | `"Domain"` or `"Service"`. |
| `EventType` | `event.EventType`. |
| `AggregateType` | CLR type name of the aggregate. |
| `AggregateBusinessKey` | `IAggregateRoot.BusinessKey`. |
| `Payload` | JSON-serialized event (web defaults). |
| `CreatedBy` | The aggregate's `ModifiedBy ?? CreatedBy` (stamped earlier by the [audit interceptor](2.AuditSaveChangesInterceptor.md)). |
| `OccurredOnUtc` | `event.OccurredOnUtc`. |
| `CreatedOnUtc` | Now (UTC). |

## Columns used by the messaging relay (not written here)

Consumers that use `RaccoonLand.Modules.Messaging.SqlServer` require two additional nullable columns on the
same table. The interceptor does **not** set them (they stay `NULL` on insert):

| Column | Role |
| --- | --- |
| `ClaimedOnUtc` | Set when a relay worker claims the row (lease). |
| `ProcessedOnUtc` | Set when dispatch/publish succeeds. |

See Messaging docs: `Modules/Messaging/SqlServer/docs/2.Schema.md`.

## Why clear waits for commit

`SavedChangesAsync` runs **before** `CommitAsync`. Clearing there would lose events if commit fails and the
execution strategy retries. On commit, only written `EventId`s are removed so an event raised after the INSERT
but before commit is not discarded. Retry also resets flush marks from `BaseCommandDbContext` per attempt, not
only via `TransactionRolledBack`. See
[ambiguous commit](1.BaseCommandDbContext.md#resiliency-caveats-ambiguous-commit) — retry is not claimed to be
safe for every commit failure mode.

## Ordering with the audit interceptor

**Required:** [`AuditSaveChangesInterceptor`](2.AuditSaveChangesInterceptor.md) must run on `SavingChanges`
before this interceptor reads `ModifiedBy` / `CreatedBy`.
[`AddRaccoonLandCommandInterceptors`](7.Registration.md) registers them in that order — do not reorder.
