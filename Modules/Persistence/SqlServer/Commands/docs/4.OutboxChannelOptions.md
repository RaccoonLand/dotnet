# 4. OutboxChannelOptions

`OutboxChannelOptions` describes **where** a registered outbox channel's rows are written on SQL Server. It is a
SQL Server / relational concern and therefore lives in this implementation package — not in the storage-agnostic
[`Outbox.Abstraction`](../../Outbox/docs/1.IOutbox.md). A different backend (Oracle, a queue, …) would define
its own channel options.

- **Package:** `RaccoonLand.Modules.Persistence.SqlServer.Commands`
- **Namespace:** `RaccoonLand.Modules.Persistence.SqlServer.Commands.Outbox`

## Surface

```csharp
public sealed class OutboxChannelOptions
{
    public string? Database { get; set; }       // null => current database
    public string Schema { get; set; } = "dbo";
    public string Table { get; set; } = string.Empty;
    public string QualifiedTableName { get; }   // [Db].[Schema].[Table] or [Schema].[Table]
}
```

| Member | Description |
| --- | --- |
| `Database` | Optional database name. When set, the table is referenced with three-part naming so the write stays inside the same transaction as `SaveChanges` (no DTC). When null, the current database is used. |
| `Schema` | Schema name (default `dbo`). |
| `Table` | Table name (required — set per channel). |
| `QualifiedTableName` | Bracket-quoted name used in the INSERT statement. |

## Same-instance constraint

The outbox write happens with Dapper on the **same connection and transaction** as EF Core's `SaveChanges`.
The target table must therefore be reachable via two- or three-part naming on the **same SQL Server instance**.
Cross-instance / distributed-transaction (DTC) scenarios are intentionally unsupported.

## Example

```csharp
services.AddRaccoonLandOutbox<IQueueOutbox>(options =>
{
    options.Schema = "messaging";
    options.Table = "QueueOutbox";
    // options.Database = "SideEffects"; // optional, same instance
});
```
