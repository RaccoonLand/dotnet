# 1. BaseCommandDbContext

`BaseCommandDbContext` is the base class for the **command-side** (write) `DbContext`. It wraps
`SaveChangesAsync` in a transaction so the [outbox interceptor](3.OutboxSaveChangesInterceptor.md) can write
aggregate events on the **same connection and transaction**, keeping the entity changes and their events
atomic.

- **Package:** `RaccoonLand.Modules.Persistence.SqlServer.Commands`
- **Namespace:** `RaccoonLand.Modules.Persistence.SqlServer.Commands`

## Responsibilities

| Concern | Behaviour |
| --- | --- |
| Event exclusion | `DomainEvent` / `ServiceEvent` and the `DomainEvents` / `ServiceEvents` collections on `IAggregateRoot` are ignored in the EF model — they are persisted via the outbox, not as relational entities. |
| Transactional save | `SaveChangesAsync` owns a transaction (unless the caller already started one) so the outbox write is committed atomically with the entity changes. |
| Connection resiliency | The transaction runs through `Database.CreateExecutionStrategy()`, so it is safe under SqlServer retry-on-failure. |
| Sync save blocked | `SaveChanges()` throws `NotSupportedException` to enforce the async, transactional path. |

## Transaction flow

```csharp
public override Task<int> SaveChangesAsync(CancellationToken cancellationToken = default)
    => SaveChangesInternalAsync(acceptAllChangesOnSuccess: true, cancellationToken);
```

- If `Database.CurrentTransaction` is **not null**, the caller owns the transaction; the context just saves
  inside it and the caller commits.
- Otherwise the context starts its own transaction, saves (the outbox interceptor's `SavedChangesAsync` runs
  here), and commits — all inside the execution strategy.

## Usage

Derive your write context and map aggregates as usual:

```csharp
public sealed class ShopCommandDbContext(DbContextOptions<ShopCommandDbContext> options)
    : BaseCommandDbContext(options)
{
    public DbSet<Product> Products => Set<Product>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        base.OnModelCreating(modelBuilder); // keep event exclusion
        modelBuilder.ApplyConfigurationsFromAssembly<ICommandEntityConfiguration>(
            typeof(ShopCommandDbContext).Assembly);
    }
}
```

Always override `OnModelCreating` by calling `base.OnModelCreating(modelBuilder)` first so event exclusion is
preserved. Use
[`RaccoonLand.Modules.Persistence.EntityFrameworkCore`](../../EntityFrameworkCore/docs/4.ApplyConfigurationsFromAssembly.md)
to apply only command-side `IEntityTypeConfiguration<>` classes from the assembly.

See also: [AuditSaveChangesInterceptor](2.AuditSaveChangesInterceptor.md),
[OutboxSaveChangesInterceptor](3.OutboxSaveChangesInterceptor.md), [Registration](4.Registration.md).
