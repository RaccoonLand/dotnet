# 1. BaseCommandDbContext

`BaseCommandDbContext` is the base class for the **command-side** (write) `DbContext`. It wraps
`SaveChangesAsync` in a transaction so outbox interceptors can write on the **same connection and
transaction**, keeping entity changes and outbox rows atomic.

- **Package:** `RaccoonLand.Modules.Persistence.SqlServer.Commands`
- **Namespace:** `RaccoonLand.Modules.Persistence.SqlServer.Commands`

## Responsibilities

| Concern | Behaviour |
| --- | --- |
| Event exclusion | `DomainEvent` / `ServiceEvent` and the `DomainEvents` / `ServiceEvents` collections on `IAggregateRoot` are ignored in the EF model — they are persisted via the outbox, not as relational entities. |
| Transactional save | `SaveChangesAsync` owns a transaction (unless the caller already started one) so the outbox write is committed atomically with the entity changes. |
| Connection resiliency | The owned-transaction path runs through `Database.CreateExecutionStrategy()`. |
| AcceptAllChanges timing | Owned path uses `acceptAllChangesOnSuccess: false` and calls `AcceptAllChanges()` only **after** a successful `CommitAsync`, so a failed commit can retry with dirty entity state. |
| In-memory outbox cleanup | On commit, only **written** event ids are removed from aggregates; the writer drops only flushed messages. |
| Attempt reset | Each execution-strategy attempt calls an explicit reset at start and on failure so retry does not depend solely on `TransactionRolledBack`. |
| Sync save blocked | `SaveChanges()` throws `NotSupportedException`. Only `SaveChangesAsync` is supported (outbox interceptors override the async path). |

## Owned transaction flow (default)

When `Database.CurrentTransaction` is null:

1. `CreateExecutionStrategy().ExecuteAsync(...)`
2. `BeginTransactionAsync`
3. `SaveChangesAsync(acceptAllChangesOnSuccess: false)` — outbox `SavedChangesAsync` INSERTs on this transaction; in-memory events/buffer are **not** cleared yet
4. `CommitAsync` — fires `TransactionCommitted` → clear events / flushed writer messages
5. `ChangeTracker.AcceptAllChanges()`

If `CommitAsync` fails, the strategy retries the whole block. Entity state is still dirty. Flush marks are
reset **explicitly** at the start of each attempt (and again in the attempt `catch`) so writer messages and
event-id marks are restored even when the provider does not raise `TransactionRolledBack`.

## Resiliency caveats (ambiguous commit)

Within a single successful round-trip, entity changes and outbox INSERTs share one transaction — that
**atomicity** holds. Execution-strategy retry after a clear failure (rollback / connection drop before commit)
is what this type optimizes for.

What this module **does not** claim: that retry is safe for every possible `CommitAsync` failure mode.
In particular, an **ambiguous commit** can occur when the database commits successfully but the client never
receives the success response (network blip after commit). The strategy may then retry the whole attempt and
re-send entity changes and outbox rows. That is a general limitation of retry-on-commit, not something this
package can eliminate in code.

Mitigations expected of the host / schema (not optional for production):

| Mitigation | Why |
| --- | --- |
| Stable unique outbox message ids | Domain/service rows use `EventId`; message-outbox rows use `Guid.CreateVersion7()` assigned at enqueue — DB unique constraints / idempotent consumers must treat duplicates as no-ops. |
| Appropriate entity primary/unique keys | Retried inserts or updates must not corrupt aggregates (concurrency tokens, natural keys). |
| Failure tests after commit, before client response | Exercise ambiguous-commit scenarios in integration tests where feasible. |

Documented guarantee: **same-transaction atomicity** for a completed save. Not documented: **fully safe retry
under every commit error**, including ambiguous commit.

## External transaction flow

When the caller already has `Database.CurrentTransaction`:

- This type only calls `base.SaveChangesAsync` inside that transaction.
- **Caller responsibilities:**
  - Commit the transaction after a successful save.
  - Rollback must undo both EF writes and Dapper outbox INSERTs (same connection/transaction).
  - Do not dispose the transaction before commit/rollback completes.
  - Execution-strategy retry across `SaveChanges` + `Commit` is **not** managed by this module — the caller owns resiliency.
- In-memory outbox cleanup still runs on `TransactionCommitted` when the caller commits.

## Usage

```csharp
public sealed class ShopCommandDbContext(DbContextOptions<ShopCommandDbContext> options)
    : BaseCommandDbContext(options)
{
    public DbSet<Product> Products => Set<Product>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        base.OnModelCreating(modelBuilder); // keep event exclusion
        modelBuilder.ApplyConfigurationsFromAssembly<ICommandEntityConfiguration>(
            typeof(ShopCommandDbContext).Assembly);
    }
}
```

Always call `base.OnModelCreating(modelBuilder)` first. Use
[`RaccoonLand.Modules.Persistence.EntityFrameworkCore`](../../EntityFrameworkCore/docs/4.ApplyConfigurationsFromAssembly.md)
for command-side configurations.
