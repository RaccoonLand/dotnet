# 3. Pagination (Queryable extensions)

EF Core helpers for the common read-side pagination pattern used with
[`PagedQuery<TResponse>`](../../../Core/RequestProcessing/RequestProcessing.Abstractions/docs/9.Pagination.md)
and [`PageResponse<TItem>`](../../../Core/RequestProcessing/RequestProcessing.Abstractions/docs/9.Pagination.md).

- **Package:** `RaccoonLand.Modules.Persistence.SqlServer.Queries`
- **Namespace:** `RaccoonLand.Modules.Persistence.SqlServer.Queries`

This package does **not** reference the CQRS abstractions. Endpoints map `PagedList<T>` to their
`PageResponse<TItem>` type.

## Types

```csharp
public readonly record struct PageRequest(int Page, int PageSize, bool IncludeTotalCount);

public sealed record PagedList<T>
{
    public IReadOnlyList<T> Items { get; init; } = [];
    public int? TotalCount { get; init; }
    public int Page { get; init; }
    public int PageSize { get; init; }
}
```

`PageRequest` can be constructed manually, but **`ApplyPaging` requires `Page ≥ 1` and `PageSize ≥ 1`**. Prefer
`Normalize` or `ToPagedListAsync`, which always produce a valid request.

## Extensions

```csharp
public static class QueryablePaginationExtensions
{
    public static PageRequest Normalize(
        int page, int pageSize, bool includeTotalCount, int maxPageSize = 100);

    public static IQueryable<T> ApplyPaging<T>(this IQueryable<T> query, PageRequest request);

    public static Task<PagedList<T>> ToPagedListAsync<T>(
        this IQueryable<T> query,
        int page,
        int pageSize,
        bool includeTotalCount = false,
        int maxPageSize = 100,
        CancellationToken cancellationToken = default);
}
```

| Method | Purpose |
| --- | --- |
| `Normalize` | Clamps `page` to at least `1` and `pageSize` to `[1, maxPageSize]`. Throws if `maxPageSize < 1`. |
| `ApplyPaging` | Applies `Skip`/`Take`. Validates page/size; uses `checked` arithmetic so an overflowing offset throws instead of wrapping. |
| `ToPagedListAsync` | Optional `CountAsync`, then fetches the page via `Normalize` + `ApplyPaging`. |

## Usage

Apply filters, **stable ordering**, and projection on the query first, then paginate. Prefer a primary sort plus
a unique tie-breaker (for example `ThenBy(x => x.Id)`) so pages do not skip or duplicate rows when values collide:

```csharp
var paged = await query
    .Where(x => x.Status == status)
    .OrderBy(x => x.LastName)
    .ThenBy(x => x.Id)
    .Select(x => new PersonListItem { Id = x.Id, LastName = x.LastName })
    .ToPagedListAsync(
        request.Page,
        request.PageSize,
        request.IncludeTotalCount,
        maxPageSize: 100,
        cancellationToken);

return Result<SearchPeopleResponse>.Success(new SearchPeopleResponse
{
    Items = paged.Items,
    TotalCount = paged.TotalCount,
    Page = paged.Page,
    PageSize = paged.PageSize,
});
```

## Rules

1. **Ordering is caller responsibility** — the extension does not impose sort order. Without a stable order
   (ideally ending in a unique key), `Skip`/`Take` can return overlapping or missing rows across pages.
2. **`IncludeTotalCount == false`** — no `CountAsync` is executed.
3. **`IncludeTotalCount == true`** — `CountAsync` runs on the filtered query before `Skip`/`Take`. That is two
   round-trips; concurrent writes can make `TotalCount` and `Items` briefly disagree. A true single snapshot is
   a consumer concern (for example a transaction/snapshot isolation) when the business requires it.
4. **`maxPageSize`** defaults to `100` and must be ≥ 1. Validators may clamp earlier; this is a safe fallback
   in the handler.
5. **No `AsNoTracking()`** — `BaseQueryDbContext` already disables tracking.
