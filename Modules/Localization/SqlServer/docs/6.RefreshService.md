# 9. MessageLocalizationRefreshService

`MessageLocalizationRefreshService` is the `BackgroundService` that keeps the [in‑memory store](4.Storage.md)
in sync with the database. It performs the initial load at startup and refreshes on a configurable interval.

- **Package:** `RaccoonLand.Modules.MessageLocalization.SQLServer`
- **Namespace:** `RaccoonLand.Modules.MessageLocalization.SQLServer.Hosting`
- **Visibility:** `internal` (registered via `AddHostedService` — see [Registration](2.Registration.md)).

## Startup (`StartAsync`)

Runs **before** the host begins serving requests, so the first lookup already sees data:

1. If `AutoCreateTables` is enabled, ensure the schema/tables exist.
2. Resolve (and optionally create) the application id via `EnsureApplicationAsync`.
3. Load the snapshot into the store.

If any step fails it is **logged as a warning** and the service starts with an empty store; the next refresh
cycle retries. Startup is never blocked by a transient database problem. A host
`OperationCanceledException` (cancellation requested) is **rethrown** and does not become a soft warning.

## Refresh loop (`ExecuteAsync`)

```csharp
using var timer = new PeriodicTimer(_options.RefreshInterval);
while (await timer.WaitForNextTickAsync(stoppingToken))
{
    // 1. (Re)resolve application id if startup failed earlier.
    // 2. Persist any reported missing keys.
    // 3. Reload the snapshot.
}
```

- A `RefreshInterval <= 0` **disables** periodic refresh (the service returns immediately).
- If the application id could not be resolved at startup (`_applicationId == 0`), the loop retries
  schema/application bootstrap before each refresh.
- Each cycle first **persists missing keys** (so newly seen keys become real placeholder rows), then
  **reloads** the snapshot so admin edits and new placeholders become visible.
- Per‑cycle failures are logged as errors and the **previous snapshot is kept** — a bad refresh never empties
  a working store.
- If persisting missing keys fails after they were drained, the batch is **requeued** onto the tracker so the
  next cycle retries; keeping the translation snapshot does not by itself preserve that batch.
- Cancellation during shutdown is handled quietly (no error noise).

## Lifecycle summary

| Phase | Action | On failure |
| --- | --- | --- |
| `StartAsync` | Ensure schema (optional) → resolve app id → initial load | Warn; start empty; retry next cycle. |
| Each tick | Persist missing keys → reload snapshot | Error; keep previous snapshot; requeue drained missing keys if persist failed. |
| Shutdown | Stop the timer loop | Cancellation swallowed. |

## Design notes

- All database work is centralized here and in the [repository](5.Repository.md), keeping the
  [localizer](3.MessageResolution.md) free of I/O.
- The "persist‑then‑reload" order means a key first requested in this process is turned into a placeholder and
  then shows up in the very next snapshot, ready for an admin to translate.
