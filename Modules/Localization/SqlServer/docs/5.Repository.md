# 8. MessageLocalizationRepository & schema

`MessageLocalizationRepository` is the Dapper‑based data access for the shared localization database. It is
used **only at startup and by the periodic refresh worker** — never on the request hot path.

- **Package:** `RaccoonLand.Modules.MessageLocalization.SQLServer`
- **Namespace:** `RaccoonLand.Modules.MessageLocalization.SQLServer.Data`
- **Visibility:** `internal`

## Data model

```
Services (1) ──< Applications (1) ──< MessageLocalizations
```

- **Services** — one row per microservice.
- **Applications** — logical applications belonging to a service.
- **MessageLocalizations** — the `(Key, Culture) → Value` entries an admin maintains at runtime.

Table and schema names are taken from [`MessageLocalizationSqlServerOptions`](1.SqlServerOptions.md). They are
**validated** as regular SQL identifiers and bracket-quoted (`[schema].[table]`, with `]` escaped). `OBJECT_ID`
checks use **parameters** for the object name (not an interpolated `N'...'` literal).

## Surface

```csharp
internal sealed class MessageLocalizationRepository
{
    public Task EnsureSchemaAsync(CancellationToken ct);
    public Task<int> EnsureApplicationAsync(CancellationToken ct);
    public Task<IReadOnlyList<LocalizationEntry>> LoadAsync(int applicationId, CancellationToken ct);
    public Task InsertMissingAsync(int applicationId, IReadOnlyCollection<MissingKey> keys, CancellationToken ct);
}
```

| Method | Description |
| --- | --- |
| `EnsureSchemaAsync` | Creates the schema and tables when missing (idempotent). Concurrent creates that hit error 2714 (object already exists) are treated as success. The `CREATE SCHEMA` identifier is built in C# and run via `sp_executesql`. |
| `EnsureApplicationAsync` | Resolves the application id for the configured service/application, creating both rows when `AutoRegisterApplication` is enabled. Concurrent inserts that hit unique violations (2601/2627) are treated as success and followed by `SELECT`. Throws if not found and auto‑register is off. |
| `LoadAsync` | Loads every `(Culture, Key, Value)` entry for the given application. |
| `InsertMissingAsync` | Persists missing keys as placeholders (`Value = Key`, `RequiresTranslation = 1`) in a single set‑based `OPENJSON` insert; existing rows are left untouched. |

## Notable implementation details

- **Single round trip for missing keys.** `InsertMissingAsync` serializes the keys to JSON and inserts them
  with one `INSERT … SELECT … FROM OPENJSON(@Json) … WHERE NOT EXISTS (…)` statement, avoiding an N+1 pattern.
- **Idempotent DDL.** Every `CREATE` is guarded by parameterized `OBJECT_ID(@name, N'U') IS NULL` /
  `sys.schemas` checks, so `EnsureSchemaAsync` is safe to re‑run.
- **Multi-replica bootstrap.** Unique constraints stay in place; a racing `INSERT` that fails with a duplicate
  key is ignored and the subsequent `SELECT` returns the winner's id.
- **No request‑path queries.** Requests are served from the [in‑memory store](4.Storage.md); the repository is
  invoked only by the [refresh worker](6.RefreshService.md).

## Schema script

A standalone, idempotent script lives at `src/SqlServer/db/MessageLocalization.Schema.sql`. Apply it once
against the shared database when you prefer to manage the schema out‑of‑band (recommended for production,
with `AutoCreateTables = false`). It creates:

- `[Localization].[Services]` — `Id`, `Name` (unique), `CreatedOnUtc`.
- `[Localization].[Applications]` — `Id`, `ServiceId` (FK), `Name`, unique `(ServiceId, Name)`.
- `[Localization].[MessageLocalizations]` — `Id`, `ApplicationId` (FK), `Key`, `Culture`, `Value`,
  `RequiresTranslation`, `CreatedOnUtc`, `ModifiedOnUtc`, unique `(ApplicationId, Key, Culture)`.

`RequiresTranslation = 1` flags entries auto‑created on a cache miss (their `Value` defaults to the key) so an
admin knows a real translation is still needed.
